planctl ÔÇö File-based task tracking for structured development workflows.

## Doc & comment style

**Never write backward-facing advice.** Docs, comments, skills, agent prompts, and diagram labels describe the system as it is *now* ÔÇö never by reference to what was removed, renamed, or previously true. No "X used toÔÇª", "formerly Y", "replaces the prior Z", "no longer carriesÔÇª", "(deleted in fn-N)", "(now removed)", or a feature defined only by negating a dead one. State the present-tense fact; if a constraint matters, phrase it as a forward rule ("there is no `--no-push` flag"), not as a tombstone. The two sanctioned exceptions are the deliberate `docs/reference/planctl-bug-history.md` record and the "Removed verbs (do not re-add)" guardrail list below ÔÇö both exist precisely to carry history.

## Convention Divergences

These diverge from standard arthack conventions:

- **`planctl cat` is format-free** ÔÇö always emits raw markdown to stdout regardless of `--format`. FormattedGroup auto-injects `--format` so the flag appears in `cat --help`, but `run_cat.py` ignores it.
- **`planctl validate` envelope is non-`success`** ÔÇö emits `{"valid": bool, "errors": [...], "warnings": [...]}` instead of the standard `{"success": bool, ...}`, exiting 1 on `valid: false`. Routes through `format_output` directly (not `emit()`) to preserve the shape.
- **Bare verb subcommand names** ÔÇö `init`, `status`, `claim`, etc. instead of arthack's `verb-noun`. Established in the spec and referenced by orchestration scripts; do not rename.

## Commit behavior

Every planctl CLI verb self-emits a `planctl_invocation` NDJSON envelope on stdout ÔÇö the wire keeper reads off `PostToolUse:Bash`. Mutating verbs additionally land their own `chore(planctl): <op> <target>` commit inline at `output.emit()` via `planctl.commit.auto_commit_from_invocation` ÔÇö there is no `commit-plan` verb, no skill-seam table, no deferral. An envelope `success: true` on stdout means the `.planctl/` commit landed. Callers never wildcard-commit planctl state themselves.

- **Mutating verbs** read the session touched-paths log (keyed by `CLAUDE_CODE_SESSION_ID`), intersect with the git dirty set to populate `files`/`subject`, then auto-commit the scoped pathspec via `git commit -F - -- <files>` (no flock; a bounded retry over git's index/ref lock domains absorbs same-host contention, re-parenting off the winner's HEAD each attempt). `task state` is now plan-state-only ÔÇö runtime status lives entirely in `.planctl/state/` (gitignored).
- **`init`** is the one mutating verb that builds its `planctl_invocation` payload directly ÔÇö no touched-paths log, no `CLAUDE_CODE_SESSION_ID` requirement. It writes a fixed bootstrap set (`.planctl/meta.json`, `.planctl/.gitignore`, `.planctl/CLAUDE.md`, the `.planctl/AGENTS.md` symlink), so its `files` is the explicit list of paths it created and the payload omits the `session_id` key (the `chore(planctl): init <project-name>` commit carries no `Session-Id:` trailer). It self-commits only when it wrote something AND the cwd is inside a git work tree; an idempotent re-run (nothing written) or an `init` in a non-git dir takes the read-only `emit()` path with no commit.
- **Read-only verbs** emit the envelope with `subject=NULL`/`files=NULL`; the auto-commit helper no-ops on empty `files`.
- **Runtime-state-only verbs** (`claim`, `block`) mutate gitignored `.planctl/state/` ÔÇö envelope carries NULL `subject`/`files`, no commit lands. `claim` additionally writes the worker brief to `<primary_repo>/.planctl/state/briefs/<task_id>.json` (inside the `lock_task` critical section) and returns a `brief_ref` handle ÔÇö the brief lives under gitignored `state/`, so this write is still commit-free.
- **`scaffold`** writes a whole epic tree in one mutating invocation but emits exactly ONE envelope and produces ONE commit covering every written path.
- **Not observable until committed.** Every mutating verb routes through the `output.emit()` central seam: emit builds the `planctl_invocation` itself, then runs the per-verb auto-commit. There is NO seam-level write-tree unwind and NO orphan reaper: a pre-commit raise (invocation-build failure or a git error) leaves any written files on disk (┬º10 no-rollback). The three multi-file mint verbs (`scaffold`, `refine-apply`, `epic create`) keep their own LOCAL write-phase try/except blocks that unwind a MID-WRITE crash (single-writer atomicity); the commit-failure window is not covered. The post-commit boundary is sacred ÔÇö once `auto_commit_from_invocation` lands, files are tracked and never unlinked (┬º10). On the consumer side, keeper's plan-worker producer gates `EpicSnapshot` / `TaskSnapshot` emission on a synchronous `git cat-file -e HEAD:<relpath>` check (the predicate lives at the plan-worker producer, NEVER in the reducer ÔÇö re-fold determinism is non-negotiable); the autopilot will not dispatch a worker against an epic whose JSON is not yet in HEAD. This keeper HEAD-gate is the SOLE guard that closes the fn-627 duplicate-dispatch window: an uncommitted epic tree on disk is harmless because it is never observed, and the next mutating verb whose `touched Ôê® dirty` intersects it sweeps it into a commit.
- **`repo_root` vs `state_repo`** in the envelope (always emitted together): `repo_root` is the cwd-derived repo at invocation; `state_repo` is where the `.planctl/` state lives (from `epic.primary_repo`, falling back to `repo_root`). `auto_commit_from_invocation` routes the commit by `state_repo`, not by cwd. Equal for single-repo projects.
- **`warnings: [str]`** rides `set-primary-repo` / `set-touched-repos` envelopes (non-blocking `_validate_repo_path` results). `scaffold` (fn-630) also surfaces an advisory entry on the success envelope `data` when an epic AND every one of its tasks ship with no snippet/bundle metadata ÔÇö the empty-shell case bundle-health flags as a no-substrate epic. Write still succeeds with exit 0 even when warnings fire ÔÇö warn-and-write is intentional. `set-target-repo` does NOT carry this field. (The scaffold advisory rides the existing `emit()` success envelope; `emit()` has no `warnings` parameter and the single-emit / single-commit contract is unchanged.)
- **`validate` is conditionally mutating** ÔÇö `validate --epic <id>` on a never-validated, valid epic writes `last_validated_at` and manually invokes `auto_commit_from_invocation` from the runner (bypassing `emit()` to preserve its `{valid, errors, warnings}` envelope shape). `validate` without `--epic`, or re-validate of a stamped epic, is read-only.
- **Commit failure** prints a structured `{"success": false, "error": "commit_failed", "details": {"error": "<code>", ...}}` envelope on stdout (exit 1) ÔÇö the success envelope is NOT printed; the structural `atomic_write`s already landed on disk. `<code>` Ôêê {`commit_contended`, `git_status`, `git_add`, `git_commit`, `missing_state_repo`, `missing_subject`}.

**Worker contract is commit-then-done**: source commit via `keeper commit-work` first (`feat(scope): ...`), then `planctl done` to stamp the task ÔÇö the `chore(planctl): done <task>` state commit lands inline as a side effect of the verb. History shows source-then-state. The worker also self-checks its own delivery cleanliness (its session files committed) before returning, so a `done` verdict from `planctl reconcile` means both commits landed and nothing of the worker's is left dirty ÔÇö the orchestrator never inspects git porcelain to attribute that.

**Source commits**: `keeper commit-work --preview-files` first, then `keeper commit-work "<msg>"`. Auto-pushes to origin on success; on push failure workers return `BLOCKED: TOOLING_FAILURE` (no inline resolve). There is no `--no-push` flag. The verb also pre-filters untracked gitignored paths (mirrors `git add .`) and hard-stops a runaway file list (> `--max-files`, default 500) with `{success:false, error:"file_list_too_large", count, limit}` ÔÇö that envelope also returns `BLOCKED: TOOLING_FAILURE` and the human re-scopes (add a `.gitignore` rule, then re-run; or pass `--max-files N` / `--max-files 0` if the count is intentional). The flock at `$GIT_COMMON_DIR/keeper-commit-work.lock` is the same-host interleave backstop for source commits; planctl's auto-commit takes no flock ÔÇö it scopes each commit to its own exact paths via `git commit -F - -- <files>` and absorbs git's index/ref lock contention with a bounded retry, so the two never cross-contaminate on disjoint pathspecs.

**Session id**: planctl resolves the session id from `CLAUDE_CODE_SESSION_ID` ÔÇö the sole source. The claude binary ships it intrinsically on every session including resumed ones (`--continue` / `--resume`); tests and manual invocations set it explicitly. There is no process-tree walk, no wildcard fallback ÔÇö `build_planctl_invocation` raises if the env var is unset. `init` is exempt: it builds its own payload directly and does NOT route through `build_planctl_invocation`, so it runs identically whether or not `CLAUDE_CODE_SESSION_ID` is set.

Full contract: `docs/reference/commit-at-mutation-boundary.md`.

## Validation marker

`normalize_epic` defaults `last_validated_at: null`. `scaffold` stamps it on fresh epic mint as part of its inline post-write integrity check ÔÇö there is no trailing `validate --epic` step after `scaffold` on either the `/plan:plan` create path or the `/plan:close` follow-up scaffold. The 14 verbs in `planctl/validation_restamp.py::VALIDATION_RESTAMP_VERBS` (canonical list ÔÇö do not duplicate here) RE-STAMP the marker to a fresh microsecond-precision `now_iso()` after their post-write integrity check passes. `epic invalidate` and `refine-context --invalidate` are the two surviving paths that null the marker (the `--invalidate` flag flips the otherwise read-only verb into a conditionally-mutating one, following `validate --epic`'s precedent). Notable non-members: `done`, `claim`, `block`, `epic close`, and `scaffold` (mints via its own path, never the restamp helper).

`validate --epic` and `refine-context --invalidate` are both conditionally-mutating verbs: when the flag is absent or the precondition is already satisfied the verb is read-only and emits no commit; when the flag fires and a write is needed, the runner manually invokes `auto_commit_from_invocation` to land a single `.planctl/` commit. `validate --epic` writes the stamp BEFORE its auto-commit (inverse of the restamp-helper shape) ÔÇö on commit failure the stamp persists on disk and a re-run short-circuits silently; the next mutating verb's auto-commit sweeps the dirty file. Full asymmetry contract: `docs/reference/commit-at-mutation-boundary.md` ┬º"validate --epic stamp-then-commit asymmetry".

## Skills and agents

`plan:*` skills live under `skills/`. Most ÔÇö including `defer` and `next` ÔÇö are hand-written tracked source skills; only `/plan:work` is generated from a template (`template/skills/work.md.tmpl`). The four worker agents live in this plugin's own `agents/` directory as generated, flat files ÔÇö `agents/worker-{medium,high,xhigh,max}.md`, each addressable `plan:worker-<tier>` ÔÇö rendered per-variant from `template/agents/worker.md.tmpl` by the `promptctl render-plugin-templates` pass (the default agents branch emits `agents/<stem>-<variant>.md` per variant; there is no `render_to:` directive and no per-tier plugin manifest). The files MUST stay flat in `agents/`: a subfolder would fold into the scoped id (yielding `plan:workers:worker-medium`). Each worker's `description` frontmatter is narrowed to internal-only so the four in-scope tiers never auto-delegate. Every rendered worker agent gets a `<name>.managed-file-dont-edit` sidecar carrying `{_warning, source_template, sha256}` that the `check-generated` guard reads (fn-637); both the rendered agents and the sidecars are gitignored ÔÇö only the template is source of truth (`agents/practice-scout.md` is the precedent). The other agents the planctl plugin owns (`quality-auditor`, `close-planner`, scouts, `gap-analyst`) stay in `agents/` too. A shared `worker_agent_for_tier(tier)` helper in `models.py` (beside `TASK_TIERS`) maps `tier ÔåÆ "plan:worker-<tier>"` (and `None ÔåÆ None`); `claim`, `worker resume`, and `resolve-task` each emit that string as a `worker_agent` envelope field. `/plan:work` is a pure pass-through: it spawns `Task(subagent_type=<worker_agent>)` straight off the envelope at both warm and cold spawn sites (no bare `work:worker` literal, no tier-picking at runtime, no `task set-tier` write), and a null-tier (legacy) task surfaces a clean typed stop at the spawn site rather than a `plan:worker-None` crash. Keeper reads `task.tier` from its own projected Task data for board/projection and launches workers with **no** `--plugin-dir` flag ÔÇö tier routing rides the emitted `worker_agent` name, not the launch line. Operational note: `CLAUDE_CODE_SUBAGENT_MODEL`, if set in the launch env, overrides every worker agent's `model` frontmatter and silently flattens all four tiers to one model. The `arthack-claude.py` launcher is generic: it does not call planctl, infer tiers, or recognize `/plan:*` prompt shapes. Load-bearing invariants:

- **`/plan:close` is a content-blind coordinator** over the `quality-auditor` and `close-planner` agents (advisory; no state mutations by the skill). The closer drives PROCESS only ÔÇö it speaks in typed envelopes and one-line agent returns and never holds the audit report, the verdict JSON, or the follow-up plan. Every pipeline artifact (audit brief, audit report, verdict, follow-up plan) persists as a file under gitignored `<primary_repo>/.planctl/state/audits/<epic_id>/`, written by the close-phase submit verbs at emission. The quality-auditor vets and persists its report via `audit submit`; the close-planner absorbs vet/cull/merge + follow-up authoring, persisting via `verdict submit` / `followup submit`. `planctl close-finalize` encodes the saga in Python from observable state ÔÇö stale-check on `commit_set_hash`, fatal-halt, reversible follow-up scaffold BEFORE the irreversible `epic close` ÔÇö and returns one of four typed `CloseOutcome` members (`closed_clean` / `closed_with_followup` / `fatal_halt` / `partial_followup`) the skill switches on totally. `close-finalize._find_followup_epic` discovers the follow-up by exact equality on the `created_by_close_of` stamp the scaffold step writes onto the minted epic (internal-only ÔÇö stamped via `getattr(args, "created_by_close_of", None)` in `run_scaffold`, never a CLI flag and never a `followup.yaml` key, so a hand-authored plan cannot spoof provenance); it never scans `depends_on_epics`, not even as a fallback, so a human-planned epic that legitimately depends on the source is invisible to the closer. The field is immutable after mint and there is no backfill of pre-fix closer-minted follow-ups. The `actual_tasks == expected` count gate is unchanged: a stamped hit with too few tasks is still `partial_followup`.
- **Inheritor skills** (`/plan:work`, `/plan:close`) push the `render-spec` cost onto a brief-handoff verb so the skill never inlines prose. On the work path `planctl claim` assembles the worker brief ÔÇö task spec, epic spec, and pre-rendered snippet context ÔÇö into `<primary_repo>/.planctl/state/briefs/<task_id>.json` and returns a `brief_ref`; `/plan:work` is content-blind and passes that handle to the worker, which reads the brief itself. On the close path `planctl close-preflight` writes the close-phase brief to `<primary_repo>/.planctl/state/audits/<epic_id>/brief.json` (commit-free, atomic) and returns `{brief_ref, commit_set_hash, primary_repo}` with no prose fields; the quality-auditor and close-planner each read the brief by path. The low-level verbs (`find-snippets`, `show-snippet`, `render`) are escape valves ÔÇö do not re-fire them in the subagents; the brief already carries the rendered context.
- **`/plan:close` runs the audit INLINE** before the irreversible `epic close` ÔÇö there is no standalone auditor session, no `auditor_done_at` stamp, no `--audit-required` flag. The planner's `fatal` flag is the only ship-block gate. `epic close` (fired by `close-finalize`) stamps `closer_done_at`, which is the completion signal keeper folds ÔÇö a closed epic is terminal and completes the instant the stamp lands.
- **`planctl resolve-task <task_id>` (fn-593)** is a read-only routing-lookup verb that returns the subset of `claim`'s envelope needed by an external launcher to route a worker and police cwd: `{task_id, epic_id, project_path, target_repo, primary_repo, tier, worker_agent, status}`. `worker_agent` is the `plan:worker-<tier>` name (via `worker_agent_for_tier`, `null` for a null-tier task) a caller spawns directly. Cwd-agnostic (scans configured `roots`, with `--project <path>` to disambiguate). Field names overlap with `claim` so a caller has one parser. Read-only ÔÇö no `.planctl/` write, no commit; envelope carries the readonly invocation footer with NULL `subject`/`files`. The `arthack-claude.py` launcher is generic and does not call this verb; keeper reads `task.tier` directly from its projected Task data for board/projection and launches workers with no `--plugin-dir` flag. The verb is a public CLI surface for other consumers.
- **Content-blind orchestrator + `planctl worker resume`** ÔÇö `/plan:work` owns PROCESS only: it speaks in typed envelopes (ids, status, shas, counts, categories) and never holds, reasons over, commits, or edits the worker's content. Every form of incompleteness the post-worker `planctl reconcile` switch surfaces ÔÇö a non-`done` `in_progress_*` / `state_uncommitted` verdict within the 5-attempt budget ÔÇö routes to one primitive: resume the worker with a minimal process nudge. The orchestrator never runs `planctl done` and never commits source on the worker's behalf, and never acts on a `tooling_error` verdict (it fails closed ÔÇö surface and stop, never resume against an unreliable verdict). `planctl worker resume <task_id>` regenerates the brief fresh (bake-fresh-on-each-entrypoint ÔÇö no provenance hashes) and returns a typed envelope `{task_id, status, tier, worker_agent, brief_ref, nudge, target_repo, primary_repo, source_commit_sha?, dirty_session_file_count?}` ÔÇö process facts only, no narrative prompt and no `planctl cat` reference. `worker_agent` is the `plan:worker-<tier>` name the orchestrator re-spawns on resume. The brief is a regeneratable cache keyed off durable runtime state; a missing or corrupt brief is re-minted by the next `claim` (ALREADY_MINE) or `worker resume`, never a fatal inconsistency.
- **`planctl reconcile <task_id>` (fn-6)** is the read-only post-worker verdict verb ÔÇö the symmetric bookend to `claim`'s pre-worker brief handoff. It collapses the `/plan:work` orchestrator's post-worker reconciliation into ONE call returning a typed verdict the orchestrator switches on mechanically. The verdict Ôêê `done | in_progress_committed | in_progress_uncommitted | blocked | state_uncommitted | not_started | tooling_error`, computed entirely from planctl-native data ÔÇö merged status (`merge_task_state`), trailer-authentic source commits (`git log` + exact-match `Task:` trailer parse against `target_repo` + `epic.touched_repos`), HEAD-visibility of the committed task JSON (`git cat-file` against `state_repo`, unborn-branch-guarded), and an epic-progress tally. NO keeper call (`dirty_session_files` attribution was explicitly rejected to keep the one-way keeperÔåÆplanctl edge ÔÇö delivery cleanliness is the worker's job), no `planctl show`, no shell-out to `planctl find-task-commit` (reconcile parses the `Task:` trailer inline), no `keeper session-state`, no `validate --epic`, no git-porcelain reasoning. Any git subprocess failure fails closed to `tooling_error` (never silently `not_started`/`done`). Read-only ÔÇö no `.planctl/` write, no commit; envelope `{verdict, task_id, epic_id, status, source_commits, state_head_visible, epic_progress, assessed_at, blocked_reason}` carries the readonly invocation footer with NULL `subject`/`files`. Cwd-agnostic (scans configured `roots`, `--project <path>` to disambiguate); bad/missing/ambiguous id ÔåÆ typed error envelope (exit 1) like `resolve-task` (`BAD_TASK_ID | TASK_NOT_FOUND | AMBIGUOUS_TASK_ID`). The orchestrator's post-worker tail is this single call + a total switch on the seven verdicts ÔÇö there is no separate orchestrator test phase, and the per-task `validate --epic` is gone; the worker reaches `done` only via a passing-test commit (commit-before-done) and self-checks its own delivery cleanliness before returning.
- **Cross-project epic deps (fn-600)** depend on the same configured `roots`. `epic.depends_on_epics` entries are bare `fn-N` ids (epic ids are globally unique via `_find_foreign_owner`) and resolve cwd-first-then-global through `planctl.discovery.resolve_epic_globally`. Without `roots`, single-repo workflows still work unchanged via the cwd short-circuit; cross-project resolution simply doesn't activate. The readiness gate distinguishes `blocked_pending` (dep resolved, not yet runtime-complete) from `blocked_dangling` (dep resolved nowhere), and the write-side surfaces `dep_ambiguous_id` instead of silently picking when a legacy dup exists. Full contract: `docs/reference/cross-project-epic-deps.md`.
- **Write-time `sketch/` inlining (fn-610 / fn-628)** ÔÇö every planctl path that writes a bundle-ref list (`scaffold`, `refine-apply`, `epic set-bundles`, `task set-bundles`) shells `promptctl inline-sketch-refs` (a thin wrapper over `promptctl.api.inline_sketch_refs`) against the cwd-derived authoring `project_root` and inlines every `sketch/<name>` ref's `snippet_ids` into the record's `snippets` list, dropping the consumed ref from `bundles`. The shared subprocess helper at `planctl/sketch_refs.py` owns the verb's argv shape (literal `["promptctl", ...]`), the per-slot error envelope (`SketchRefError(ref, reason)`), and the tooling-failure envelope (`SketchToolingError`) ÔÇö planctl carries zero in-repo Python dependency on promptctl, exactly as it already does for `promptctl render-spec`. The `@logged` audit line for `inline_sketch_refs` now fires in the promptctl subprocess (one line per scaffold spawn). Persisted epic/task records never carry a `sketch/` ref, so a worker on a different machine never tries to resolve a sketch authored under another project's gitignored `.promptctl/sketches/`. `bundle/` refs pass through unchanged. An unresolvable sketch surfaces as `ref_invalid` in the verb's assert phase; `render-spec` stays strict ÔÇö no graceful-degrade at runtime.
- **`/plan:defer <subject>` and `/plan:next <epic_id>`** split single-task scaffold and priority-flip into two hand-written tracked source skills (`name: defer`, `name: next`) ÔÇö neither is template-generated. `/plan:defer` is the sole single-task scaffolder: it mainlines the actionable conversation work into a single-task epic at normal epic-number order and stops on overrun (offering a concrete alternative, never silently scaling up). `/plan:next` operates on an *existing* epic ÔÇö it does NOT scaffold ÔÇö and flips board priority by calling `planctl epic queue-jump <epic_id>`, a conditionally-mutating epic verb with a read-only short-circuit when already set. `queue-jump` sets `queue_jump=true` and emits an envelope carrying `queue_jump: true`; keeperd's reducer derives `queue_jump` (sticky-true) from any event carrying that signal and stamps a `!`-prefixed `sort_path` so the epic sorts above all other root epics on the board. Both skills are members of the `/plan:plan` family ÔÇö NOT job-launchers, no worker spawn, no audit, no `_ROLE_SKILL_NAMES` entry, no `<role>::<id>` session-naming prefix.

For the per-spec snippet/bundle metadata model and the full design, see `~/arcs/snippeting/items/runtime-substrate-snippets-as/README.md`.

## Removed verbs (do not re-add)

`start` (ÔåÆ `claim`), `next`, `activity`, `scout`, `interview`, `config show`, `draft from-close`, `epic publish`, `epic create --draft`, `epic.draft` field, `epic auditor-done`, `migrate-audit-grandfathering`, `audit-guard`, `close-context`, `epic close --audit-required`/`--no-audit-required`, `epic.auditor_done_at` field, `task create`, `task set-spec`, `task set-deps`, `epic set-plan`, `dep add` (whole `dep` group), `gravity`, `classifier` (agent + `skills/close/classifier/` schema ÔÇö the close-planner absorbs vet/cull/merge + verdict authoring), `epic followup-of` (the partial-follow-up completeness check is ported into `close-finalize`). Their structural effects ride `scaffold` (mint a fresh epic + tasks) and `refine-apply` (add/rewrite on an existing epic). Epic-level deps keep `epic add-dep` / `epic add-deps` / `epic rm-dep`.

The no-incremental-mutation stance above is NOT a no-delete stance. `planctl epic rm <epic_id>` (fn-623) is the sanctioned ÔÇö and only ÔÇö delete verb: it unlinks every artifact an epic owns (epic JSON, every child task JSON, epic + task spec markdowns, runtime state, lock files) and auto-commits the deletions into the owning project's `.planctl/` via the standard envelope path. Resolves cwd-then-global with a `--project` escape, guards `in_progress` / locked tasks behind `--force`, and supports `--dry-run`. Companion guard on the create side: `scaffold` rejects a same-slug sibling epic up front with `duplicate_epic` (naming the existing id + status); `--allow-duplicate` is the explicit escape hatch. Together they replace the manual `find -delete` + hand surgery path that motivated the epic.
