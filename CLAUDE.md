planctl — File-based task tracking for structured development workflows.

## Doc & comment style

**Never write backward-facing advice.** Docs, comments, skills, agent prompts, and diagram labels describe the system as it is *now* — never by reference to what was removed, renamed, or previously true. No "X used to…", "formerly Y", "replaces the prior Z", "no longer carries…", "(deleted in fn-N)", "(now removed)", or a feature defined only by negating a dead one. State the present-tense fact; if a constraint matters, phrase it as a forward rule ("there is no `--no-push` flag"), not as a tombstone. The two sanctioned exceptions are the deliberate `docs/reference/planctl-bug-history.md` record and the "Removed verbs (do not re-add)" guardrail list below — both exist precisely to carry history.

## Convention Divergences

These diverge from standard arthack conventions:

- **`planctl cat` is format-free** — always emits raw markdown to stdout regardless of `--format`. FormattedGroup auto-injects `--format` so the flag appears in `cat --help`, but `run_cat.py` ignores it.
- **`planctl validate` envelope is non-`success`** — emits `{"valid": bool, "errors": [...], "warnings": [...]}` instead of the standard `{"success": bool, ...}`, exiting 1 on `valid: false`. Routes through `format_output` directly (not `emit()`) to preserve the shape.
- **Bare verb subcommand names** — `init`, `status`, `claim`, etc. instead of arthack's `verb-noun`. Established in the spec and referenced by orchestration scripts; do not rename.

## Commit behavior

Every planctl CLI verb self-emits a `planctl_invocation` NDJSON envelope on stdout — the wire keeper reads off `PostToolUse:Bash`. Mutating verbs additionally land their own `chore(planctl): <op> <target>` commit inline at `output.emit()` via `planctl.commit.auto_commit_from_invocation` — there is no `commit-plan` verb, no skill-seam table, no deferral. An envelope `success: true` on stdout means the `.planctl/` commit landed. Callers never wildcard-commit planctl state themselves.

- **Mutating verbs** read the session touched-paths log (keyed by `CLAUDE_CODE_SESSION_ID`), intersect with the git dirty set to populate `files`/`subject`, then auto-commit the scoped pathspec via `git commit -F - -- <files>` (no flock; a bounded retry over git's index/ref lock domains absorbs same-host contention, re-parenting off the winner's HEAD each attempt). `task state` is now plan-state-only — runtime status lives entirely in `.planctl/state/` (gitignored).
- **`init`** is the one mutating verb that builds its `planctl_invocation` payload directly — no touched-paths log, no `CLAUDE_CODE_SESSION_ID` requirement. It writes a fixed bootstrap set (`.planctl/meta.json`, `.planctl/.gitignore`, `.planctl/CLAUDE.md`, the `.planctl/AGENTS.md` symlink), so its `files` is the explicit list of paths it created and the payload omits the `session_id` key (the `chore(planctl): init <project-name>` commit carries no `Session-Id:` trailer). It self-commits only when it wrote something AND the cwd is inside a git work tree; an idempotent re-run (nothing written) or an `init` in a non-git dir takes the read-only `emit()` path with no commit.
- **Read-only verbs** emit the envelope with `subject=NULL`/`files=NULL`; the auto-commit helper no-ops on empty `files`.
- **Runtime-state-only verbs** (`claim`, `block`, `approve`) mutate gitignored `.planctl/state/` — envelope carries NULL `subject`/`files`, no commit lands.
- **`scaffold`** writes a whole epic tree in one mutating invocation but emits exactly ONE envelope and produces ONE commit covering every written path.
- **Not observable until committed.** Every mutating verb routes through the `output.emit()` central seam: emit builds the `planctl_invocation` itself, then runs the per-verb auto-commit. There is NO seam-level write-tree unwind and NO orphan reaper: a pre-commit raise (invocation-build failure or a git error) leaves any written files on disk (§10 no-rollback). The three multi-file mint verbs (`scaffold`, `refine-apply`, `epic create`) keep their own LOCAL write-phase try/except blocks that unwind a MID-WRITE crash (single-writer atomicity); the commit-failure window is not covered. The post-commit boundary is sacred — once `auto_commit_from_invocation` lands, files are tracked and never unlinked (§10). On the consumer side, keeper's plan-worker producer gates `EpicSnapshot` / `TaskSnapshot` emission on a synchronous `git cat-file -e HEAD:<relpath>` check (the predicate lives at the plan-worker producer, NEVER in the reducer — re-fold determinism is non-negotiable); the autopilot will not dispatch a worker against an epic whose JSON is not yet in HEAD. This keeper HEAD-gate is the SOLE guard that closes the fn-627 duplicate-dispatch window: an uncommitted epic tree on disk is harmless because it is never observed, and the next mutating verb whose `touched ∩ dirty` intersects it sweeps it into a commit.
- **`repo_root` vs `state_repo`** in the envelope (always emitted together): `repo_root` is the cwd-derived repo at invocation; `state_repo` is where the `.planctl/` state lives (from `epic.primary_repo`, falling back to `repo_root`). `auto_commit_from_invocation` routes the commit by `state_repo`, not by cwd. Equal for single-repo projects.
- **`warnings: [str]`** rides `set-primary-repo` / `set-touched-repos` envelopes (non-blocking `_validate_repo_path` results). `scaffold` (fn-630) also surfaces an advisory entry on the success envelope `data` when an epic AND every one of its tasks ship with no snippet/bundle metadata — the empty-shell case bundle-health flags as a no-substrate epic. Write still succeeds with exit 0 even when warnings fire — warn-and-write is intentional. `set-target-repo` does NOT carry this field. (The scaffold advisory rides the existing `emit()` success envelope; `emit()` has no `warnings` parameter and the single-emit / single-commit contract is unchanged.)
- **`validate` is conditionally mutating** — `validate --epic <id>` on a never-validated, valid epic writes `last_validated_at` and manually invokes `auto_commit_from_invocation` from the runner (bypassing `emit()` to preserve its `{valid, errors, warnings}` envelope shape). `validate` without `--epic`, or re-validate of a stamped epic, is read-only.
- **Commit failure** prints a structured `{"success": false, "error": "commit_failed", "details": {"error": "<code>", ...}}` envelope on stdout (exit 1) — the success envelope is NOT printed; the structural `atomic_write`s already landed on disk. `<code>` ∈ {`commit_contended`, `git_status`, `git_add`, `git_commit`, `missing_state_repo`, `missing_subject`}.

**Worker contract is commit-then-done**: source commit via `keeper commit-work` first (`feat(scope): ...`), then `planctl done` to stamp the task — the `chore(planctl): done <task>` state commit lands inline as a side effect of the verb. History shows source-then-state.

**Source commits**: `keeper commit-work --preview-files` first, then `keeper commit-work "<msg>"`. Auto-pushes to origin on success; on push failure workers return `BLOCKED: TOOLING_FAILURE` (no inline resolve). There is no `--no-push` flag. The verb also pre-filters untracked gitignored paths (mirrors `git add .`) and hard-stops a runaway file list (> `--max-files`, default 500) with `{success:false, error:"file_list_too_large", count, limit}` — that envelope also returns `BLOCKED: TOOLING_FAILURE` and the human re-scopes (add a `.gitignore` rule, then re-run; or pass `--max-files N` / `--max-files 0` if the count is intentional). The flock at `$GIT_COMMON_DIR/keeper-commit-work.lock` is the same-host interleave backstop for source commits; planctl's auto-commit takes no flock — it scopes each commit to its own exact paths via `git commit -F - -- <files>` and absorbs git's index/ref lock contention with a bounded retry, so the two never cross-contaminate on disjoint pathspecs.

**Session id**: planctl resolves the session id from `CLAUDE_CODE_SESSION_ID` — the sole source. The claude binary ships it intrinsically on every session including resumed ones (`--continue` / `--resume`); tests and manual invocations set it explicitly. There is no process-tree walk, no wildcard fallback — `build_planctl_invocation` raises if the env var is unset. `init` is exempt: it builds its own payload directly and does NOT route through `build_planctl_invocation`, so it runs identically whether or not `CLAUDE_CODE_SESSION_ID` is set.

Full contract: `docs/reference/commit-at-mutation-boundary.md`.

## Validation marker

`normalize_epic` defaults `last_validated_at: null`. `scaffold` stamps it on fresh epic mint as part of its inline post-write integrity check — there is no trailing `validate --epic` step after `scaffold` on either the `/plan:plan` create path or the `/plan:close` follow-up scaffold. The 14 verbs in `planctl/validation_restamp.py::VALIDATION_RESTAMP_VERBS` (canonical list — do not duplicate here) RE-STAMP the marker to a fresh microsecond-precision `now_iso()` after their post-write integrity check passes. `epic invalidate` and `refine-context --invalidate` are the two surviving paths that null the marker (the `--invalidate` flag flips the otherwise read-only verb into a conditionally-mutating one, following `validate --epic`'s precedent). Notable non-members: `done`, `claim`, `block`, `epic close`, ack verbs (runtime, not structural), and `scaffold` (mints via its own path, never the restamp helper).

`validate --epic` and `refine-context --invalidate` are both conditionally-mutating verbs: when the flag is absent or the precondition is already satisfied the verb is read-only and emits no commit; when the flag fires and a write is needed, the runner manually invokes `auto_commit_from_invocation` to land a single `.planctl/` commit. `validate --epic` writes the stamp BEFORE its auto-commit (inverse of the restamp-helper shape) — on commit failure the stamp persists on disk and a re-run short-circuits silently; the next mutating verb's auto-commit sweeps the dirty file. Full asymmetry contract: `docs/reference/commit-at-mutation-boundary.md` §"validate --epic stamp-then-commit asymmetry".

## Skills and agents

`plan:*` skills live under `skills/`. Most — including `defer` and `next` — are hand-written tracked source skills; only `/plan:work` is generated from a template (`template/skills/work.md.tmpl`). Worker agents do NOT live in this plugin's `agents/` anymore — they ship as separate per-tier plugins under `work-plugins/<tier>/agents/worker.md`, rendered from `template/agents/worker[-codex].md.tmpl` via the cross-boundary `render_to:` directive (fn-593). Each tier's `.claude-plugin/plugin.json` manifest is ALSO generated by the same `promptctl render-plugin-templates` pass — its name/description come from the template's `manifest_description:` frontmatter field (Jinja-rendered with `{{ current_variant }}`), and every rendered output (agent + manifest) gets a `<name>.managed-file-dont-edit` sidecar carrying `{_warning, source_template, sha256}` that the `check-generated` guard reads (fn-637). Both the manifests and the sidecars are gitignored — only the templates are source of truth. Other agents the planctl plugin still owns (`classifier`, `quality-auditor`, scouts, `gap-analyst`) stay in `agents/`. Keeper reads `task.tier` from its own projected Task data and launches workers with `--plugin-dir <planctl_root>/work-plugins/<tier>` on the claude launch line, so exactly one `work:worker` agent is in scope per session — `/plan:work` spawns `Task(subagent_type="work:worker")` as a bare literal (no tier-picking at runtime, no `task set-tier` write). The `arthack-claude.py` launcher is generic: it does not call planctl, infer tiers, or recognize `/plan:*` prompt shapes. Load-bearing invariants:

- **`/plan:close` uses the `quality-auditor` agent** for its inline audit (advisory; no state mutations).
- **Inheritor skills** (`/plan:work`, `/plan:close`) call `promptctl render-spec <id>` ONCE at startup and pipe the prose into their subagent brief under `## Snippet context`. The low-level verbs (`find-snippets`, `show-snippet`, `render`) are escape valves — do not re-fire them in the subagent; the parent already paid the render cost.
- **`/plan:close` runs the audit INLINE** before the irreversible `epic close` — there is no standalone auditor session, no `auditor_done_at` stamp, no `--audit-required` flag. `epic close` stamps only `closer_done_at`; the epic flips to `pending_approval` on the human ack.
- **`planctl resolve-task <task_id>` (fn-593)** is a read-only routing-lookup verb that returns the subset of `claim`'s envelope needed by an external launcher to pick a tier-plugin and police cwd: `{task_id, epic_id, project_path, target_repo, primary_repo, tier, status}`. Cwd-agnostic (scans configured `roots`, with `--project <path>` to disambiguate). Field names overlap with `claim` so a caller has one parser. Read-only — no `.planctl/` write, no commit; envelope carries the readonly invocation footer with NULL `subject`/`files`. The `arthack-claude.py` launcher is generic and does not call this verb; keeper reads `task.tier` directly from its projected Task data and launches with model/effort/name/plugin-dir itself. The verb is a public CLI surface for other consumers.
- **Cross-project epic deps (fn-600)** depend on the same configured `roots`. `epic.depends_on_epics` entries are bare `fn-N` ids (epic ids are globally unique via `_find_foreign_owner`) and resolve cwd-first-then-global through `planctl.discovery.resolve_epic_globally`. Without `roots`, single-repo workflows still work unchanged via the cwd short-circuit; cross-project resolution simply doesn't activate. The readiness gate distinguishes `blocked_pending` (dep resolved, not yet runtime-complete) from `blocked_dangling` (dep resolved nowhere), and the write-side surfaces `dep_ambiguous_id` instead of silently picking when a legacy dup exists. Full contract: `docs/reference/cross-project-epic-deps.md`.
- **`/plan:approve <epic_id|task_id>` (fn-592)** is the human-approval skill. Keeper launches approve workers under sonnet+low (passed via `--model sonnet --effort low` on the launcher command line) with a `--name approve::<id>` slug. The skill reads a keeperd render of the target's context and writes the decision via the gated `planctl approve <id> [status]` CLI verb. `derive_role` returns `approver` for these invocations; session-naming format `approve::<id>` is rendered upstream by keeper, not by the launcher or the session-namer hook (see `apps/hookctl/CLAUDE.md` § Session naming).
- **Approval is a gitignored runtime sidecar (fn-732).** `approval` lives canonically in `.planctl/state/` — the `approval` key on the per-task state file (`state/tasks/<id>.state.json`) and a per-epic sidecar (`state/epics/<id>.state.json`) — so keeper folds it gate-free. `planctl approve` is **runtime-state-only**: it writes ONLY the sidecar (task path is a `lock_task` RMW that preserves a concurrent `status` write) and emits a read-only invocation (NULL `subject`/`files`), so no `.planctl/` commit lands — it mirrors `claim`/`block`. Every approval reader resolves the value via the ladder **sidecar → committed def → `pending`**; the `pending` default lives in the merge step (`merge_task_state` / `merge_epic_state`), never in `normalize_*`. The committed-def rung is keeper's permanent fallback for the keeper-boots-first race and for legacy defs written before approval moved to the sidecar; the one-shot `scripts/migrate_approval_to_sidecar.py` seeds every existing def's approval into the sidecar (idempotent, fill-if-absent) and — gated on a positive end-to-end sidecar-fold verify — strips `approval` from the def files (`--strip`).
- **Write-time `sketch/` inlining (fn-610 / fn-628)** — every planctl path that writes a bundle-ref list (`scaffold`, `refine-apply`, `epic set-bundles`, `task set-bundles`) shells `promptctl inline-sketch-refs` (a thin wrapper over `promptctl.api.inline_sketch_refs`) against the cwd-derived authoring `project_root` and inlines every `sketch/<name>` ref's `snippet_ids` into the record's `snippets` list, dropping the consumed ref from `bundles`. The shared subprocess helper at `planctl/sketch_refs.py` owns the verb's argv shape (literal `["promptctl", ...]`), the per-slot error envelope (`SketchRefError(ref, reason)`), and the tooling-failure envelope (`SketchToolingError`) — planctl carries zero in-repo Python dependency on promptctl, exactly as it already does for `promptctl render-spec`. The `@logged` audit line for `inline_sketch_refs` now fires in the promptctl subprocess (one line per scaffold spawn). Persisted epic/task records never carry a `sketch/` ref, so a worker on a different machine never tries to resolve a sketch authored under another project's gitignored `.promptctl/sketches/`. `bundle/` refs pass through unchanged. An unresolvable sketch surfaces as `ref_invalid` in the verb's assert phase; `render-spec` stays strict — no graceful-degrade at runtime.
- **`/plan:defer <subject>` and `/plan:next <epic_id>`** split single-task scaffold and priority-flip into two hand-written tracked source skills (`name: defer`, `name: next`) — neither is template-generated. `/plan:defer` is the sole single-task scaffolder: it mainlines the actionable conversation work into a single-task epic at normal epic-number order and stops on overrun (offering a concrete alternative, never silently scaling up). `/plan:next` operates on an *existing* epic — it does NOT scaffold — and flips board priority by calling `planctl epic queue-jump <epic_id>`, a conditionally-mutating epic verb with a read-only short-circuit when already set. `queue-jump` sets `queue_jump=true` and emits an envelope carrying `queue_jump: true`; keeperd's reducer derives `queue_jump` (sticky-true) from any event carrying that signal and stamps a `!`-prefixed `sort_path` so the epic sorts above all other root epics on the board. Both skills are members of the `/plan:plan` family — NOT job-launchers, no worker spawn, no audit, no `_ROLE_SKILL_NAMES` entry, no `<role>::<id>` session-naming prefix.

For the per-spec snippet/bundle metadata model and the full design, see `~/arcs/snippeting/items/runtime-substrate-snippets-as/README.md`.

## Removed verbs (do not re-add)

`start` (→ `claim`), `next`, `activity`, `scout`, `interview`, `config show`, `draft from-close`, `epic publish`, `epic create --draft`, `epic.draft` field, `epic auditor-done`, `migrate-audit-grandfathering`, `audit-guard`, `close-context`, `epic close --audit-required`/`--no-audit-required`, `epic.auditor_done_at` field, `task create`, `task set-spec`, `task set-deps`, `epic set-plan`, `dep add` (whole `dep` group), `gravity`. Their structural effects ride `scaffold` (mint a fresh epic + tasks) and `refine-apply` (add/rewrite on an existing epic). Epic-level deps keep `epic add-dep` / `epic add-deps` / `epic rm-dep`.

The no-incremental-mutation stance above is NOT a no-delete stance. `planctl epic rm <epic_id>` (fn-623) is the sanctioned — and only — delete verb: it unlinks every artifact an epic owns (epic JSON, every child task JSON, epic + task spec markdowns, runtime state, lock files) and auto-commits the deletions into the owning project's `.planctl/` via the standard envelope path. Resolves cwd-then-global with a `--project` escape, guards `in_progress` / locked tasks behind `--force`, and supports `--dry-run`. Companion guard on the create side: `scaffold` rejects a same-slug sibling epic up front with `duplicate_epic` (naming the existing id + status); `--allow-duplicate` is the explicit escape hatch. Together they replace the manual `find -delete` + hand surgery path that motivated the epic.
