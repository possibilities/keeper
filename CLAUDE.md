planctl — File-based task tracking for structured development workflows.

## Convention Divergences

These diverge from standard arthack conventions:

- **`planctl cat` is format-free** — always emits raw markdown to stdout regardless of `--format`. FormattedGroup auto-injects `--format` so the flag appears in `cat --help`, but `run_cat.py` ignores it.
- **`planctl validate` envelope is non-`success`** — emits `{"valid": bool, "errors": [...], "warnings": [...]}` instead of the standard `{"success": bool, ...}`, exiting 1 on `valid: false`. Routes through `format_output` directly (not `emit()`) to preserve the shape.
- **Bare verb subcommand names** — `init`, `status`, `claim`, etc. instead of arthack's `verb-noun`. Established in the spec and referenced by orchestration scripts; do not rename.

## Commit behavior

Every planctl CLI verb self-emits a `planctl_invocation` NDJSON envelope on stdout — the wire keeper reads off `PostToolUse:Bash`. Mutating verbs additionally land their own `chore(planctl): <op> <target>` commit inline at `output.emit()` via `planctl.commit.auto_commit_from_invocation` — there is no `commit-plan` verb, no skill-seam table, no deferral. An envelope `success: true` on stdout means the `.planctl/` commit landed. Callers never wildcard-commit planctl state themselves.

- **Mutating verbs** read the session touched-paths log (keyed by `PLANCTL_SESSION_ID`), intersect with the git dirty set to populate `files`/`subject`, then auto-commit the scoped pathspec under planctl's own `$GIT_COMMON_DIR/planctl-commit.lock` flock. `task state` is now plan-state-only — runtime status lives entirely in `.planctl/state/` (gitignored).
- **Read-only verbs** emit the envelope with `subject=NULL`/`files=NULL`; the auto-commit helper no-ops on empty `files`.
- **Runtime-state-only verbs** (`claim`, `block`) mutate gitignored `.planctl/state/` — envelope carries NULL `subject`/`files`, no commit lands.
- **`scaffold`** writes a whole epic tree in one mutating invocation but emits exactly ONE envelope and produces ONE commit covering every written path.
- **Not observable until committed (fn-629).** As of fn-629 tasks .1–.4 every mutating verb routes through the `output.emit()` central seam: emit owns the write→commit transaction, builds the `planctl_invocation` itself, and unwinds the caller's `written_paths` on any pre-commit raise — so a failed multi-file verb (`scaffold`, `refine-apply`, `epic create`, `epic rm`) leaves zero on-disk side effects. Single-field seam-routed verbs pass `written_paths=[]` (rename-atomic over tracked files; prior valid contents stay in place on a pre-commit raise). The post-commit boundary is sacred — once `auto_commit_from_invocation` lands, files are tracked and never unlinked (§10). On the consumer side, keeper's plan-worker producer gates `EpicSnapshot` / `TaskSnapshot` emission on a synchronous `git cat-file -e HEAD:<relpath>` check (the predicate lives at the plan-worker producer, NEVER in the reducer — re-fold determinism is non-negotiable); the autopilot will not dispatch a worker against an epic whose JSON is not yet in HEAD. The pair (commit-at-the-seam; refuse-to-observe-pre-commit) closes the fn-627 duplicate-dispatch window end-to-end. A hard `commit_failed` from an unmigrated path (or a future runaway) that leaves a fully-written but untracked epic tree on disk is reaped automatically: `planctl.reaper.reap_orphan_epics` runs as a Phase-3 pre-flight in `scaffold` / `refine-apply` and unlinks any orphan tree whose epic JSON is git-untracked AND mtime-stale (>5min) AND host-wide-unowned (no touched-paths-log activity within 10min), making §10's "next mutating verb sweeps it" promise real for the keeper-gated disjoint case. The reaper is fail-soft (a sweep error never blocks the actual mutation) and conservatively skips fresh in-flight pre-commit trees so the fn-627 window stays safe.
- **`repo_root` vs `state_repo`** in the envelope (always emitted together): `repo_root` is the cwd-derived repo at invocation; `state_repo` is where the `.planctl/` state lives (from `epic.primary_repo`, falling back to `repo_root`). `auto_commit_from_invocation` routes the commit by `state_repo`, not by cwd. Equal for single-repo projects.
- **`warnings: [str]`** rides `set-primary-repo` / `set-touched-repos` envelopes (non-blocking `_validate_repo_path` results). `scaffold` (fn-630) also surfaces an advisory entry on the success envelope `data` when an epic AND every one of its tasks ship with no snippet/bundle metadata — the empty-shell case bundle-health flags as a no-substrate epic. Write still succeeds with exit 0 even when warnings fire — warn-and-write is intentional. `set-target-repo` does NOT carry this field. (The scaffold advisory rides the existing `emit()` success envelope; `emit()` has no `warnings` parameter and the single-emit / single-commit contract is unchanged.)
- **`validate` is conditionally mutating** — `validate --epic <id>` on a never-validated, valid epic writes `last_validated_at` and manually invokes `auto_commit_from_invocation` from the runner (bypassing `emit()` to preserve its `{valid, errors, warnings}` envelope shape). `validate` without `--epic`, or re-validate of a stamped epic, is read-only.
- **Commit failure** prints a structured `{"success": false, "error": "commit_failed", "details": {"error": "<code>", ...}}` envelope on stdout (exit 1) — the success envelope is NOT printed; the structural `atomic_write`s already landed on disk. `<code>` ∈ {`lock_timeout`, `git_status`, `git_add`, `git_commit`, `missing_state_repo`, `missing_subject`}.

**Worker contract is commit-then-done**: source commit via `jobctl commit-work` first (`feat(scope): ...`), then `planctl done` to stamp the task — the `chore(planctl): done <task>` state commit lands inline as a side effect of the verb. History shows source-then-state.

**Source commits**: `jobctl commit-work --preview-files` first, then `jobctl commit-work "<msg>"`. Auto-pushes to origin on success; on push failure workers return `BLOCKED: TOOLING_FAILURE` (no inline resolve). There is no `--no-push` flag. The verb also pre-filters untracked gitignored paths (mirrors `git add .`) and hard-stops a runaway file list (> `--max-files`, default 500) with `{success:false, error:"file_list_too_large", count, limit}` — that envelope also returns `BLOCKED: TOOLING_FAILURE` and the human re-scopes (add a `.gitignore` rule, then re-run; or pass `--max-files N` / `--max-files 0` if the count is intentional). The flock at `$GIT_COMMON_DIR/jobctl-commit.lock` is the same-host interleave backstop for source commits; planctl's auto-commit uses its own independent flock at `$GIT_COMMON_DIR/planctl-commit.lock` (the two target disjoint pathspecs and do not need to serialize against each other).

**Session id**: planctl resolves the session id from `PLANCTL_SESSION_ID` — the sole source. The claude launcher exports it on every spawned session; tests and manual invocations set it explicitly. There is no process-tree walk, no wildcard fallback — `build_planctl_invocation` raises if the env var is unset.

Full contract: `docs/reference/commit-at-mutation-boundary.md`.

## Validation marker

`normalize_epic` defaults `last_validated_at: null`. `scaffold` stamps it on fresh epic mint as part of its inline post-write integrity check — there is no trailing `validate --epic` step after `scaffold` on either the `/plan:plan` create path or the `/plan:close` follow-up scaffold. The 14 verbs in `planctl/validation_restamp.py::VALIDATION_RESTAMP_VERBS` (canonical list — do not duplicate here) RE-STAMP the marker to a fresh microsecond-precision `now_iso()` after their post-write integrity check passes. `epic invalidate` and `refine-context --invalidate` are the two surviving paths that null the marker (the `--invalidate` flag flips the otherwise read-only verb into a conditionally-mutating one, following `validate --epic`'s precedent). Notable non-members: `done`, `claim`, `block`, all `set-*-review-status`, `epic close`, ack verbs (runtime, not structural), and `scaffold` (mints via its own path, never the restamp helper).

`validate --epic` and `refine-context --invalidate` are both conditionally-mutating verbs: when the flag is absent or the precondition is already satisfied the verb is read-only and emits no commit; when the flag fires and a write is needed, the runner manually invokes `auto_commit_from_invocation` to land a single `.planctl/` commit. `validate --epic` writes the stamp BEFORE its auto-commit (inverse of the restamp-helper shape) — on commit failure the stamp persists on disk and a re-run short-circuits silently; the next mutating verb's auto-commit sweeps the dirty file. Full asymmetry contract: `docs/reference/commit-at-mutation-boundary.md` §"validate --epic stamp-then-commit asymmetry".

## Skills and agents

`plan:*` skills live under `skills/` (some gitignored, generated from `skill-templates/*.tmpl`). Worker agents do NOT live in this plugin's `agents/` anymore — they ship as separate per-tier plugins under `work-plugins/<tier>/agents/worker.md`, rendered from `agent-templates/worker[-codex].md.tmpl` via the cross-boundary `render_to:` directive (fn-593). Other agents the planctl plugin still owns (`classifier`, `quality-auditor`, scouts, `gap-analyst`) stay in `agents/`. Keeper reads `task.tier` from its own projected Task data and launches workers with `--plugin-dir <planctl_root>/work-plugins/<tier>` on the claude launch line, so exactly one `work:worker` agent is in scope per session — `/plan:work` spawns `Task(subagent_type="work:worker")` as a bare literal (no tier-picking at runtime, no `task set-tier` write). The `arthack-claude.py` launcher is generic: it does not call planctl, infer tiers, or recognize `/plan:*` prompt shapes. Load-bearing invariants:

- **`/plan:review-*` skills are never auto-invoked and have no auto-loop** — slash-only, single backend (Codex). `review-quality` / `close` use the `quality-auditor` agent (advisory; no state mutations).
- **Inheritor skills** (`/plan:work`, `/plan:close`, `/plan:audit`, `/plan:review-*`) call `promptctl render-spec <id>` ONCE at startup and pipe the prose into their subagent brief under `## Snippet context`. The low-level verbs (`find-snippets`, `show-snippet`, `render`) are escape valves — do not re-fire them in the subagent; the parent already paid the render cost.
- **`/plan:close` runs the audit INLINE** before the irreversible `epic close` — there is no standalone auditor session, no `auditor_done_at` stamp, no `--audit-required` flag. `epic close` stamps only `closer_done_at`; the epic flips to `pending_approval` on the human ack.
- **`planctl resolve-task <task_id>` (fn-593)** is a read-only routing-lookup verb that returns the subset of `claim`'s envelope needed by an external launcher to pick a tier-plugin and police cwd: `{task_id, epic_id, project_path, target_repo, primary_repo, tier, status}`. Cwd-agnostic (scans configured `roots`, with `--project <path>` to disambiguate). Field names overlap with `claim` so a caller has one parser. Read-only — no `.planctl/` write, no commit; envelope carries the readonly invocation footer with NULL `subject`/`files`. The verb is no longer wired to the `arthack-claude.py` launcher (which is now a generic wrapper — keeper reads `task.tier` directly from its projected Task data and launches with model/effort/name/plugin-dir itself). The verb is retained as a public CLI surface for other consumers.
- **Cross-project epic deps (fn-600)** depend on the same configured `roots`. `epic.depends_on_epics` entries are bare `fn-N` ids (epic ids are globally unique via `_find_foreign_owner`) and resolve cwd-first-then-global through `planctl.discovery.resolve_epic_globally`. Without `roots`, single-repo workflows still work unchanged via the cwd short-circuit; cross-project resolution simply doesn't activate. The readiness gate distinguishes `blocked_pending` (dep resolved, not yet runtime-complete) from `blocked_dangling` (dep resolved nowhere), and the write-side surfaces `dep_ambiguous_id` instead of silently picking when a legacy dup exists. Full contract: `docs/reference/cross-project-epic-deps.md`.
- **`/plan:approve <epic_id|task_id>` (fn-592)** is the human-approval skill that replaces the prior agent-plugin + producer-pipeline approach (stripped in this epic). Keeper launches approve workers under sonnet+low (passed via `--model sonnet --effort low` on the launcher command line) with a `--name approve::<id>` slug. The skill reads a keeperd render of the target's context and writes the decision via the gated `planctl approve <id> [status]` CLI verb (renamed from the prior ungated verb). `derive_role` returns `approver` for these invocations; session-naming format `approve::<id>` is rendered upstream by keeper, not by the launcher or the session-namer hook (see `apps/hookctl/CLAUDE.md` § Session naming).
- **Write-time `sketch/` inlining (fn-610 / fn-628)** — every planctl path that writes a bundle-ref list (`scaffold`, `refine-apply`, `epic set-bundles`, `task set-bundles`) shells `promptctl inline-sketch-refs` (a thin wrapper over `promptctl.api.inline_sketch_refs`) against the cwd-derived authoring `project_root` and inlines every `sketch/<name>` ref's `snippet_ids` into the record's `snippets` list, dropping the consumed ref from `bundles`. The shared subprocess helper at `planctl/sketch_refs.py` owns the verb's argv shape (literal `["promptctl", ...]`), the per-slot error envelope (`SketchRefError(ref, reason)`), and the tooling-failure envelope (`SketchToolingError`) — planctl carries zero in-repo Python dependency on promptctl, exactly as it already does for `promptctl render-spec`. The `@logged` audit line for `inline_sketch_refs` now fires in the promptctl subprocess (one line per scaffold spawn). Persisted epic/task records never carry a `sketch/` ref, so a worker on a different machine never tries to resolve a sketch authored under another project's gitignored `.promptctl/sketches/`. `bundle/` and `arc/` refs pass through unchanged. An unresolvable sketch surfaces as `ref_invalid` in the verb's assert phase; `render-spec` stays strict — no graceful-degrade at runtime.
- **`/plan:queue <subject>` and `/plan:defer <subject>` (fn-595)** are sibling single-task scaffolders generated from one shared template at `skill-templates/queue-defer.md.tmpl` (`Generated by:` header). Both mainline into a single-task epic and stop on overrun (offering a concrete alternative, never silently scaling up). They are members of the `/plan:plan` family — NOT job-launchers, no worker spawn, no audit, no `_ROLE_SKILL_NAMES` entry, no `<role>::<id>` session-naming prefix. `/plan:queue` additionally sets `queue_jump: true` on the `planctl_invocation` envelope (server-derived from the scaffold event, immutable across event-log replay); keeperd lifts it into the `epics.queue_jump` projection column (schema v30) and stamps a `!`-prefixed `sort_path` so queue-jumped epics sort above all other root epics on the board. `queue_jump` is set skill-side, not via a CLI flag — there is no `--queue-jump` argument on `planctl scaffold`.

For the per-spec snippet/bundle metadata model and the full design, see `~/arcs/snippeting/items/runtime-substrate-snippets-as/README.md`.

## Removed verbs (do not re-add)

`start` (→ `claim`), `next`, `activity`, `scout`, `interview`, `config show`, `draft from-close`, `epic publish`, `epic create --draft`, `epic.draft` field, `epic auditor-done`, `migrate-audit-grandfathering`, `audit-guard`, `close-context`, `epic close --audit-required`/`--no-audit-required`, `epic.auditor_done_at` field, `task create`, `task set-spec`, `task set-deps`, `epic set-plan`, `dep add` (whole `dep` group), `gravity`. Their structural effects ride `scaffold` (mint a fresh epic + tasks) and `refine-apply` (add/rewrite on an existing epic). Epic-level deps keep `epic add-dep` / `epic add-deps` / `epic rm-dep`.

The no-incremental-mutation stance above is NOT a no-delete stance. `planctl epic rm <epic_id>` (fn-623) is the sanctioned — and only — delete verb: it unlinks every artifact an epic owns (epic JSON, every child task JSON, epic + task spec markdowns, runtime state, lock files) and auto-commits the deletions into the owning project's `.planctl/` via the standard envelope path. Resolves cwd-then-global with a `--project` escape, guards `in_progress` / locked tasks behind `--force`, and supports `--dry-run`. Companion guard on the create side: `scaffold` rejects a same-slug sibling epic up front with `duplicate_epic` (naming the existing id + status); `--allow-duplicate` is the explicit escape hatch. Together they replace the manual `find -delete` + hand surgery path that motivated the epic.
