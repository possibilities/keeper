## Overview

Make the `/plan:work` orchestrator content-blind. Today it holds the full epic spec, task spec, and rendered snippet prose in its own LLM context only to string-substitute them into the worker spawn — pure pass-through it never reasons over. This epic moves that brief out-of-band into a gitignored file: `planctl claim` writes `<primary_repo>/.planctl/state/briefs/<task_id>.json` and returns a `brief_ref` handle instead of inlining the three prose fields; the worker reads the file itself. The orchestrator owns PROCESS and speaks only in typed envelopes (ids, status, shas, counts, categories); the worker owns CONTENT, commits, delivery, and self-diagnosing what's left on resume. Every form of incompleteness (harness drop, dirty-after-done) routes to one primitive: resume the worker with a minimal process nudge — the orchestrator never commits, never runs `planctl done`, never edits.

Design pinned in a `/hack` session with codex (gpt-5.5) review. Option B (per-task generated agent bound via `--plugin-dir`) was considered and rejected — it forces keeper to make plugin-dir first-class per-task job metadata across both dispatch and resume. Keeper is UNTOUCHED by this epic (it does not parse the claim envelope; it reads `task.tier` from its own projection).

## Best practices

- **Atomic brief write:** same-directory temp → `os.replace` (not `os.rename`); reuse `store.atomic_write_json` for the atomic-replace baseline. Set mode `0600` and avoid following symlinks on create; clean up the temp file on write failure. [practice-scout: filelock CVE-2025-68146, LWN atomic-write]
- **Integer `schema_version` starting at 1:** a reader that sees `schema_version` greater than it knows hard-fails (`brief_schema_too_new`); additive-only changes within v1. [practice-scout: Solace schema-registry]
- **Repair-on-reclaim:** the brief is a regeneratable cache, the runtime state is the durable source of truth; a missing/corrupt brief is regenerated by the next `claim` (ALREADY_MINE) or `worker resume`, never a fatal inconsistency. [practice-scout]
- **Proportionality:** the brief consumer is a trusted local LLM worker reading via the Read tool, not a hostile Python `open()` — write-side hygiene (atomicity, 0600, no-symlink-follow on create, confinement under `state/briefs/`) is what matters; do not over-engineer reader-side TOCTOU hardening.
