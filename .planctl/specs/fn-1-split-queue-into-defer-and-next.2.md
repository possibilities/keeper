## Description

**Size:** M
**Files:** skills/next/SKILL.md (new), tests/test_next_skill_consistency.py (new)

Add a hand-written `/plan:next` skill that marks an existing epic to jump
the board queue by calling the `epic queue-jump` verb from task 1. It does
NOT scaffold — it operates on an epic that already exists.

### Approach

Author `skills/next/SKILL.md` as a tracked source skill (no template, no
`Generated by:` header). Frontmatter: `name: next`, a description with
triggers ("next", "do this next", "jump the queue", "prioritize", "top of
the board", `/plan:next`), `argument-hint: "[epic_id]"`,
`allowed-tools: Bash(planctl:*), Read`. Auto-invokable (no
`disable-model-invocation`), mirroring the old queue skill's stance.

Body flow:
- Resolve the epic id: if `$ARGUMENTS` is a `fn-N-slug` epic id, use it.
  Else INFER from the conversation — specifically scan for the
  most-recent `planctl scaffold` success envelope's `epic_id` (the
  just-minted epic). This is the INVERSE of defer's guard: defer excludes
  `.planctl/`-sourced content, but `/plan:next` WANTS the just-created
  epic, so the scaffold envelope is the legitimate source. Echo the
  resolved id in italics and block on ack before mutating.
- Run `planctl epic queue-jump <epic_id>` and report the result (the
  new sort position / queue-jump status) in one line.
- Guardrails: reject a task id (`fn-N-slug.M`) — queue-jump is epic-level;
  direct to the parent epic. If no id is given and none can be inferred,
  ask for one rather than guessing.

Look at `skills/approve/SKILL.md` and the current `skills/defer/SKILL.md`
for the source-skill frontmatter shape and the echo-then-ack idiom.
