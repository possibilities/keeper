# CONTEXT — keeper ubiquitous language
## Event-sourcing core
- **Event / Synthetic event**: An Event is an immutable, totally-ordered source-of-truth record; a Synthetic event is one keeper mints so an internal mutation round-trips through that same stream. Avoid: message, log entry, command, direct write.
- **Projection / Fold**: A Projection is a disposable read view derived from the event stream. A Fold is the pure step that applies one Event to it and advances the Cursor in the same transaction. Avoid: cache, table of record, ingest, handler.
- **Reducer**: The component that folds events into projections and owns the sole write path to them. Avoid: processor, service, controller.
- **Cursor**: The last event id a reducer has folded, marking how far a projection has caught up; a stream position, distinct from the per-row lifecycle stamp. Avoid: pointer, offset, watermark.
- **Lifecycle stamp**: The per-job event-time high-water mark that lifecycle state transitions may not regress behind, so a stale out-of-order event annotates but never resurrects state. Avoid: watermark, cursor, last-seen.
- **Re-fold**: Rebuilding a projection by replaying every event, which stays deterministic only because a fold never reads wall-clock, environment, or the filesystem. Avoid: rebuild, replay-repair, reprocess.
- **Dead letter**: An event the reducer could not fold, parked for inspection and later replay instead of crashing the fold; a **poison row** is one whose payload failed classification, blocking the commit rail within its evidence scope until an operator resolve or current-parser re-classification. Avoid: error queue, reject, bad message.
- **Live-only projection**: A projection derived from the live world rather than replayed events, so it is refreshed in place and never wiped by a migration rewind; its producer may retire an individual row it declares gone via a targeted tombstone. Avoid: snapshot, ephemeral view, scratch state.
- **Migration ladder**: The ordered array of explicit-version `{version, kind, apply}` step entries `migrate()` applies in order, with `SCHEMA_VERSION` derived as the tail entry's version rather than hand-typed; a one-lane-at-a-time singleton resource, so two concurrent schema edits collide at merge rather than compose silently. Avoid: registry, migration list, schema chain.
## Plan board
- **Board**: The read-only plan state — epics, tasks, and their readiness — that an agent orients on before acting. Avoid: backlog, kanban, queue.
- **Epic**: A tracked unit of work holding a spec and an ordered set of dependent tasks. Avoid: project, story, milestone.
- **Task**: One acceptance-bounded slice of an epic that a single worker implements end-to-end. Avoid: subtask, issue, chore.
- **Plan**: The durable spec-and-dependency graph for an epic, authored interactively and consumed read-only, never mutated by the reducer. Avoid: roadmap, schedule, spec sheet.
- **Id ledger**: The host-local append-only record of every plan number a project has handed out, consulted at mint alongside the directory scan so destroying a minted file can never free its number. Avoid: high-water mark, watermark, counter.
- **Brief**: The self-contained context packet a worker receives for its task, carrying the spec and glossary out-of-band instead of inlined prose. Avoid: prompt, ticket body, handoff note.
- **Readiness**: The gate deciding whether a task may dispatch, recomputed each cycle from its dependencies and validation state. Avoid: status, priority, availability.
- **Arm**: To flip an epic or task from not-ready to dispatchable by stamping it validated. Avoid: enable, approve, unlock.
- **Ghost**: A not-yet-validated epic or task that renders dashed and blocks dispatch until it is armed. Avoid: draft, stub, placeholder.
- **Tier**: The capability class assigned to a task that selects which model and worker cell runs it. Avoid: level, rank, weight.
- **Worker cell**: The one `{model × tier}` `work` plugin a task's launcher selects at launch; a native cell runs its model in-session, a wrapped cell delegates to the model's serving provider. Avoid: variant, flavor, profile.
- **Cell assignment / Dispatched cell**: A Cell assignment is the committed, receipt-backed `{model, tier}` choice for the exact current todo set; an unassigned task carries null axes and cannot arm or dispatch. The Dispatched cell records the provider-equivalence translation actually launched at claim, clearing for a later unconstrained attempt; absent means the assigned cell ran. Avoid: provisional default, placeholder cell, suggested cell, effective cell, executed cell, override cell.
- **Capability model**: The model-axis value a task carries, a capability token derived from a Provider's launch id (the segment after its last `/`, the whole id when slash-free); native when the claude provider serves it, wrapped otherwise. Avoid: harness model, backend model, model alias.
- **Launch id**: A Provider's model entry, the string a harness CLI receives verbatim — a bare scalar or an `{id, efforts}` form carrying a per-model effort-list override — from which the Capability model derives by basename. Avoid: alias target, model alias, native alias.
- **Supported harness**: One of Claude or Pi, the closed set Keeper may launch, resume, restore, capture, or register as a Provider; vendor or model words inside a Launch id never change its harness. Avoid: known harness, provider family, model vendor.
- **Provider**: A Supported harness's entry in the host matrix config, tying it to the launch ids it serves. Avoid: vendor, backend, platform.
- **Pecking order**: The provider list order in the host matrix config, cost-ascending, deciding at run time which provider serves a wrapped cell; the same derived capability served by more than one provider is one axis value, owned by the first provider in the order, with every other entry recorded as a shadow — tracked in the parsed matrix, not yet surfaced on any reading path. Avoid: priority list, fallback chain, ranking.
- **Agent pin**: A static subagent's `{model, effort}` row in the host matrix's `agent_pins` map, baked into the agent's frontmatter at render time; a pair, never a triple, because frontmatter is consumed in-session by the harness. Avoid: frontmatter default, static value, model override.
- **Wrapped cell**: A worker cell whose model claude does not serve natively; its worker is a claude wrapper that delegates implementation to the model's provider and owns the keeper close-out. Avoid: foreign cell, proxy worker, delegated task.
- **Wrapper driver**: The fixed claude model-and-effort every wrapped cell's wrapper runs at, set in the host matrix config. Avoid: chaperone, host model.
- **Provider leg**: The Harness session a wrapped cell's wrapper launches to implement the task, grouped with its peers in the shared `wrapped` Tmux session; its task-ID Session title is display metadata, while its run handle and Harness resume target own waiting, cleanup, and continuation. Avoid: panel leg, wrapped worker, delegate session.
- **Provider-leg result envelope**: The uniform `{schema_version, agent, handle, transcript_path, resume_target, message, message_found, elapsed_seconds, outcome}` object `keeper agent run`/`wait` write to the marker's `KEEPER_WRAPPED_ENVELOPE` path on every leg outcome, the wrapper's sole read of the provider leg's result. Avoid: leg output, run result, delegate response.
- **Provider-leg death notice / abort capture**: A death notice is a size-bounded, versioned Agent Bus artifact keyed by the leg's authoritative terminal event, routed best-effort to its one live wrapper attempt while folded lifecycle pull remains truth. An abort capture is the redacted, byte-bounded pane scrollback plus structured exit signal/code a producer stamps on a wrapped leg's terminal Killed event when it dies before writing a transcript, so a pre-boot death stays attributable; a typed capture-unavailable marker stands in when the pane is already gone. Avoid: ownership edge, stop event, durable lifecycle truth, crash log, stderr dump, core dump.
- **Provider-leg ownership / Provider-leg-active**: Provider-leg ownership durably binds a Provider leg to the exact wrapper Dispatch attempt that launched it, authorizing terminal-cascade teardown and exact claim release; survival across owners takes one fenced transfer transition. Provider-leg-active is the readiness reason when wrapper-side child evidence is age-stale but a directly owned live leg has fresh activity in the same window, so the row remains running and occupies the mutex. Avoid: ownership edge (the death-notice non-synonym), title inference, adoption, sub-agent-running, leg heartbeat, fresh wrapper.
- **Leg cascade**: The level-triggered producer that, on an owning wrapper attempt's folded terminal or superseded transition, identity-rechecks and TERM→KILLs every leg it still owns, confirms each exit from folded evidence, then releases that attempt's exact claims. Avoid: reaper (the stuck-work sweeps), autoclose, kill sweep.
- **Partner-died outcome**: The typed `partner_died` run-capture result when folded lifecycle proves the exact Provider leg terminal without a fresh settled stop; callers resume or relaunch instead of waiting on the dead handle. Avoid: timeout, no-message, launch failure.
- **Baseline**: The daemon-computed suite result at a commit sha that a worker consults to attribute a test failure as pre-existing or self-inflicted. Avoid: cache, snapshot, golden.
- **Blocking follow-up / Close-phase artifact / Resume grade**: A Blocking follow-up must complete before its source epic may stamp done, holding every dependent; a Close-phase artifact is persisted report, verdict, follow-up, or selection evidence whose Resume grade compares it with the fresh commit set, so a redispatched close skips valid phases while staleness invalidates it and downstream phases. Avoid: gating epic, close blocker, receipt, restore, harness resume, resume cursor.
## Personal notes
- **Note / Disposition**: A Note is a durable user-authored text capture that stays active until its Disposition — the successful clipboard copy or fresh-agent launch that processes it — archives it to browsable history. Avoid: snippet (a prompt-corpus entry), task, handoff, send, archive (the resulting state), delivery.
## Autopilot and dispatch
- **Autopilot / Reconciler**: Autopilot is the server-side loop that dispatches ready tasks and closes finished epics without a human; its Reconciler is the level-triggered core that re-derives what should be running each cycle. Avoid: scheduler, cron, orchestrator, dispatcher, poller, event loop.
- **Dispatch**: To fire one worker at a task or a close, whether by autopilot or by hand. Avoid: launch, spawn, enqueue.
- **Harness activity**: The derived state of one Harness session — active while its main turn or an attributable work-bearing child is live, quiescent on positive idle evidence, and unknown when evidence proves neither; ambient infrastructure never makes it active. Avoid: job state, pane liveness, slot.
- **Dispatch attempt**: One monotonically ordered, fenced try to bind a Dispatch target to a Harness session, independent of tmux Generation and process identity. Avoid: generation, retry, launch.
- **Parked launch**: A fired Dispatch attempt whose pending record remains unbound to any Harness session beyond the launch grace; it raises a self-clearing distress naming the inspectable Tmux window and clears on a late bind. Avoid: parked Dispatch claim, idle session, dead launch, pending task.
- **Dispatch claim / Orphaned claim**: A Dispatch claim is the durable exclusive right of one attempt to bind, resume, or release a target; an Orphaned claim is exact and acquired but has no bound Harness session or surviving pending launch row, so the boot-anchored Reaper expires it visibly before ownerless release. Avoid: occupancy, slot, parked claim, stale pending, phantom worker.
- **Resource hold**: The exact-identity guard preventing pane, window, lane, worktree, or cwd destruction while a Harness session may still use it, independent of Harness activity and a Dispatch claim. Avoid: workspace lease, occupancy, cleanup lock.
- **Dispatch table**: The launch catalog's per-verb `dispatch:` map assigning each dispatched session's launch triple, floored to the compiled-in constants when a row is absent or the catalog is unreadable. Avoid: role key, worker preset, escalation preset.
- **Equivalence map**: The committed directional mapping naming, for every dispatchable worker cell of one provider family, its most-equivalent cell in the other; the two directions are authored independently and are never inverses. Avoid: alias map, translation table, model crosswalk.
- **Provider constraint**: The durable work-dispatch policy requiring each selected Worker cell to run through one provider family; it translates cross-family assignments through the Equivalence map, leaves same-family assignments intact, and admits only an effective cell with a compatible launch route, driver, marker, and manifest. Avoid: provider pin, Agent pin, model override, fallback preference.
- **Reaper**: A background sweep that reclaims stuck, stale, or dead work so the board keeps moving. Avoid: cleanup job, garbage collector, timeout.
- **Phantom-working / Zombie session**: Phantom-working is a job row stuck reading working after its session has gone permanently idle, so autoclose, readiness, and dependent dispatches all consume the wrong state. A Zombie session is a stopped job whose recorded process is still alive with no activity evidence — often holding a done-stamped task's completion hostage; the hybrid reaper kills only the done-stamped case and pages the rest. Avoid: zombie job, ghost worker, stale running, orphan process, lingering agent.
- **Drain**: Folding a backlog of pending events to completion in bounded batches. Avoid: flush, catch-up, backfill.
- **Sticky / Operator jam**: A Sticky is a dispatch failure that stays parked and visible until an operator retries it rather than clearing itself; an Operator jam is its non-self-clearing reason class, which wedges the board and fires alarm surfaces while the broad sticky count remains a status display. Avoid: transient error, flake, warning, stuck row, hard failure, blockage.
- **Needs-human**: The family of board signals requiring operator attention — dead letters, block escalations, unanswered questions, and stuck dispatches, with finalize-non-ff, the instant-death wall, and homed blocked-work rows as subsets of stuck dispatches that never double-count into the total, plus display-only members — the pinned epic and the finalize-pending count (a done epic whose lane has not landed while the board is paused) — that never count toward the jam total at all. Avoid: alert queue, error state, attention list.
- **Pinned epic**: The full epic block a live close/work dispatch failure keeps rendered on the board after the epic closes — display-only, its lifetime exactly the failure row's. Avoid: sticky epic, ghost epic, zombie row.
- **Selection review**: The committed, per-epic dataset of out-of-band verdicts grading whether the cell each task actually ran — its Dispatched cell when recorded, else its Cell assignment — was underpowered, right-sized, or overpowered; a human-invoked skill assembles and grades it after the epic closes, and it is advisory input to model-selector policy, never a live board signal. Avoid: selection score, rating, quality audit.
- **Selector verdict**: The raw JSON cell-set a model-selector subagent returns as its final message — untrusted input the calling skill pipes verbatim to the trusted apply seam, which stages it as the ordinal-keyed verdict document under gitignored selection state for close-finalize to consume. Avoid: assignment, selection (alone), verdict file.
- **Model card**: The cached, provenance-headed, sha256-pinned vendor system-card markdown for one model-selector research entry, converted from the vendor's own capability doc and required for that model to classify `fresh`; distinct from the research notes `reference`, which is the skill-authored distilled behavioral cache the card's raw signal feeds. Avoid: reference (that names the notes file), spec sheet, datasheet.
- **Instant-death wall**: The needs-human threshold reached when enough dispatch keys trip the instant-death breaker at once that the failures read as an account or quota wall rather than isolated crashes. Avoid: crash wall, breaker count, death spiral.
- **Distress row**: A sticky daemon-minted needs-human signal keyed in dispatch_failures, cleared only by its own positive level-trigger (a recovered boot rate, a drained evidence window), never by retry. Avoid: alert, log line, exception.
- **Escalation dispatch**: The autopilot response to a stuck task or merge that fires a purpose-built session carrying an assembled incident brief, rather than waking the work's original creator. Bounded by a global concurrent cap on turn-active occupancy (a stopped-with-idle-backend session does not count as live) and per-epic serialization; the human is paged exactly once, only when a session declines or dies. Avoid: creator-wake, planner-notify, page.
- **Unblock session**: The `unblock::<task>` escalation session autopilot dispatches for a blocked task whose category is escalatable, loading the brief to resolve the block without the creator's context. Distinct from the `keeper plan unblock` board verb (which flips a task `blocked → todo`): the session is the escalation actor, the verb is one action it may take. Avoid: unblock verb, resume, re-dispatch.
- **Deconflict session**: The `deconflict::<epic>` (or task-scoped `deconflict::<taskId>`) escalation session autopilot dispatches once the tier-1 Resolver declines a fan-in merge conflict, loading the context the mechanical resolver lacked to settle a conflict that needs judgment. Avoid: resolver, merger, second-pass resolver.
- **Repair session / SHARED_BASE_BROKEN**: SHARED_BASE_BROKEN is the baseline-gated blocked category for a repo's default branch red at HEAD in a healthy environment, independent of a worker's diff; autopilot routes it to a write-capable, repo-scoped `repair::<repo>` Repair session that lands a shared-checkout trunk commit and fans out unblock to every affected task, carrying a bounded failing-tests digest and the baseline leaf key as its diagnosis. A repair left stopped or dead without a recorded terminal outcome past a grace classifies as declined and pages once. Avoid: deconflict session, base fixer, trunk bot, broken build, base failure, pre-existing failure.
- **Block instance**: One entering-blocked episode of a task, identified by the event that armed it; an unblock followed by a re-block opens a new instance, and escalation state is scoped per instance, never per task. Avoid: block, blocked state, re-block (as a noun for the same instance).
- **Pill**: A small colored status badge the board and statusline render to show a job's kind or state. Avoid: label, tag, icon.
- **Per-root cap**: The per-repo concurrent-dispatch limit stored as durable intent, applying while worktree mode is on. The effective cap dispatch honors is derived at read time and floors to one whenever worktree mode is off. Avoid: pinned cap; conflating stored with effective.
## Worktree and merge
- **Lane**: A per-task worktree the autopilot derives from the dependency graph each cycle to run tasks in parallel. Avoid: branch, checkout, slot.
- **Worktree mode**: A producer-only autopilot setting that gives each ready task its own isolated checkout instead of sharing one. Avoid: parallel mode, multi-branch, fork mode.
- **Merge-gate**: The check that holds a dependent lane until every upstream it needs has truly merged into the local default branch. Avoid: barrier, dependency wait, lock.
- **Base-drift / Base-freshness**: Base-drift measures a lane's divergence from local default by behind-count and merge-base age. Base-freshness is the producer-only, threshold-and-quiescence-gated refresh that merges local default into that lane on a bounded cadence. Avoid: staleness, branch lag, rebase debt, lane reset.
- **Resolver**: The tier-1 autonomous worker autopilot dispatches to settle a mechanically-clear fan-in merge conflict, epic-scoped or task-scoped to match the conflicting row, stamping blocked when judgment is needed; a decline hands the conflict off to the context-loaded Deconflict session. Avoid: merger, rebase bot, fixer, deconflict session.
- **Fan-in**: The convergence point where parallel lanes or failures collapse onto one epic- or task-keyed outcome. Avoid: join, merge point, aggregation.
- **Recover pass**: The per-cycle worktree sweep that aborts interrupted merges, merges a done-but-unmerged epic base into the default branch, and retires dead lanes — backing a closed epic's un-tearable lane's dirt up to the lane dirt spool then force-removing it past a grace, while a foreign, ambiguous, or locked lane is never destroyed and pages once instead. Avoid: cleanup pass, reconcile, gc.
- **Lane dirt spool**: The env-overridable state directory where a lane's — or, via the dead-writer sweep, a shared checkout's — tracked diffs and untracked files are snapshotted immediately before any force teardown or clean; a failed snapshot blocks the destroy. Avoid: backup dir, stash, trash.
- **Dead-writer sweep**: The producer pass that backs up and cleans shared-checkout dirt only when every session whose cwd is that checkout is provably terminal past a grace and no merge is in progress, letting the dirty row's level-clear observe the clean tree; any ambiguity pages once instead. Avoid: owner sweep, dirt GC, auto-clean.
- **Durable await**: A persisted await intent serviced by a leased daemon worker — surviving the arming session, evaluating only server-evaluable conditions, and firing its follow-up as a fresh session at most once — as opposed to the ephemeral in-process await that dies with its client. Avoid: background await, watch, subscription.
- **Lane pre-merge**: The guard that vets a dependent task's base lane BEFORE its fan-in merges the completed siblings in — restoring a provably-redundant leak to the base's HEAD, deferring a base it cannot safely settle to a self-clearing row, and escalating a persistent wedge to a needs-human distress; distinct from the fan-in conflict itself, which a Resolver settles rather than self-clearing. Avoid: clean, cleanup, premerge fixup, fan-in conflict.
## Commit ownership and publication
- **Exclusive file claim**: Exact mutation evidence that may automatically select a dirty path for its owning invocation — direct tool, Plan, or synchronous direct evidence — and whose live or unknown foreign owner blocks adoption. Avoid: touch, mention, shell evidence, dirty file.
- **Mutation observation**: Non-exclusive evidence that a Bash command, generator, package manager, or time bracket may explain a dirty path; it is an adoption hint and never automatic ownership. Avoid: claim, owner, attribution (unqualified).
- **Invocation adoption**: A caller's one-invocation decision to include exact live-dirty paths lacking automatic ownership, refused by any live or unknown foreign Exclusive file claim and creating no durable claim. Avoid: force, takeover, ownership transfer, allowlist.
- **Vacated claim**: An Exclusive file claim whose claimant the pid-and-start-time witness proves gone, making the path adoptable without erasing durable ownership evidence. Avoid: abandoned claim, stolen ownership, forced adoption.
- **Voluntary release record**: A claimant-authored, identity-proven, size-bounded record voluntarily giving named paths back to a blocked peer while the claimant stays alive; commit-work reads it as a live sibling of the Vacated claim's gone-witness, and it self-fences its own writer's later claims over those paths. Avoid: forced release, forfeiture, lease release.
- **Request-release notice**: The refusal envelope's typed pointer naming a live claimant and contested paths plus a `keeper session release` invocation, delivered as one bounded, best-effort bus send that is never load-bearing — an inbox-less or unresponsive claimant cannot deadlock the requester. Avoid: signal, page, demand.
- **Receipts-pending**: The typed refusal when un-ingested receipts are the only blocker to an otherwise-terminal foreign adoption, carrying bounded lag and stalled-ingester evidence. Avoid: ownership conflict, retryable ambiguity, daemon lag.
- **Frozen commit identity**: The sorted selected path set bound to each path's Git-normalized kind, blob OID, and mode plus the resulting private-index tree, which may not refresh during lint, hooks, signing, or publication. Avoid: staged set, pathspec, snapshot, file list.
- **Atomic commit publication**: Creating and verifying a commit from the Frozen commit identity, then compare-and-swap advancing exactly the captured branch from its captured parent; a ref race refuses rather than rolls another writer back. Avoid: commit command, path-limited commit, best-effort publish.
- **Attribution floor**: The per-root event-id fence represented by the latest live Git status or one impossible-path row while dropped, compacting discharged path history and preventing old mutation events from resurrecting. Avoid: global Git floor, Cursor, tombstone set.
## Account routing
- **Capacity observation / Account route / Usage meter / Measurement age / Usage view / Provider account category / Capacity multiplier**: A Capacity observation is freshness-bounded claude-swap advice whose Account route is one new Claude process's managed `claude-swap:<slot>` path; its freshness is the advice's age, while Measurement age is provider-reported Usage meter provenance, never eligibility policy. The Usage view renders decision meters and explicitly segregated last-good display meters that never authorize an Account route, plus optional PII-free Provider account categories (bounded canonical subscription classes) and Capacity multipliers (explicit numeric allowance classes); unknown values are omitted, never inferred. Avoid: usage model, scraper row, quota projection, profile, pin, affinity, native route, raw provider payload, treating Measurement age as observation freshness, Tier (the task capability term), plan name, guessed multiplier, arbitrary provider text.
- **Account focus / Fable focus / Non-Fable focus / Fable intent**: An Account focus is durable operator intent scoped to Fable or proven non-Fable Claude work, preferring one eligible Account route and falling back visibly to normal Account route selection; a Fable focus also soft-avoids its target for non-Fable work when no matching focus applies. Fable intent is the process-lineage routing classification that lets a continuation preserve that scope without creating account affinity. Avoid: drain (the event-fold term), account pin, sticky account, conversation affinity.
- **Launch attribution / Launch reservation**: Launch attribution immutably records which Account route one process used but never routes later work; a Launch reservation is short-lived non-exclusive pressure preventing concurrent launch stampedes. Avoid: account affinity, lease, lock, ownership.
- **Codex account alias / Codex account pool / Codex capacity observation / Codex quota scope / Codex session route / Routing pressure / Substantive output / Proof window / Genuine proof / Forced-refresh seam / Fault-injection seam**: An alias is an opaque, non-PII Provider id routed by freshness-bounded observations; its quota scope is exactly `generic` or `model:gpt-5.3-codex-spark`, derived from the requested model. Usage, stickiness, quota cooldown, and authorization are scope-local, while pressure and auth/rate/transport cooldowns are account-wide; Substantive output closes replay, and genuine proof requires an armed window's recorded refresh and fault paths. Avoid: display label, provider meter text, email, profile, credential farm, account affinity, lease, synthetic proof, manually asserted report, refresh mock, fake provider.
## Bus, presence, and session surface
- **Agent Bus**: The local message bus running agents use to talk to each other, joined by subscribing a watch channel. Avoid: pubsub, chat room, socket.
- **Bus message artifact**: The private immutable file carrying one Agent Bus message's content while the bus carries only its confined reference; it remains readable through ordinary message retention and any queued-for-wake lifetime. Avoid: spill file, attachment, inline payload.
- **Presence**: Being a live participant on the bus by holding an open watch subscription, not merely having sent a message. Avoid: online status, heartbeat, session.
- **Tmux session**: The terminal-multiplexer container workers, viewers, and panels launch into; an unqualified "session" in a launch or dispatch context means this one. Avoid: workspace, window group, terminal.
- **Harness session**: One persisted agent conversation owned by a Supported harness, with its own transcript and immutable harness-native id; jobs and forensics may refer to it, while its display title is never identity. Avoid: job, chat, conversation.
- **Claude session**: A Harness session owned by Claude Code. Avoid: using it for a Pi conversation.
- **Session title / Canonical slug / Session reference**: A Session title is a human-renamable lookup alias, never identity or a resume key; a Canonical slug is Keeper's lowercase ASCII letter/digit/hyphen form, normalized from inferred text and required verbatim by Pi's explicit `/rename`, while Claude explicit names follow Claude Code's native syntax. A Session reference selects a qualified native id, Keeper job alias, or exact current or historical title; distinct matches remain ambiguous. Avoid: session name, label, fuzzy name, sanitized title, session identity.
- **Session catalog**: The unified read model of discoverable Harness sessions, joining transcript-native identity and content with optional Keeper job aliases, title history, and lifecycle facts without requiring a session to be Keeper-tracked. Avoid: jobs list, transcript registry, session store.
- **History index**: A private, disposable search derivative of Harness session content that is always rebuildable from native artifacts and never a source of truth. Avoid: event Projection, archive, transcript store, sidecar.
- **File evidence**: A provenance-graded relationship between a Harness session and a path: observed mutation, possible mutation, or textual mention; a weaker grade never implies a stronger one. Avoid: treating mention, attempted tool use, or shell inference as a confirmed edit.
- **Transcript reader**: The per-harness discovery-and-parse module behind session traversal, owning where that harness's sessions live on disk and folding them into one harness-neutral transcript model; the transcript registry's key set is its own membership root, deliberately narrower than the launchable harness list. Avoid: transcript watcher (the run-capture stop-scan), transcript worker (the statusline tail-watch), parser.
- **Sidecar**: The private per-turn docs mirror a session maintains alongside its work, owned by hooks and never the doc body itself. Avoid: backup, shadow copy, cache.
- **Board-work session**: A session keeper itself dispatched — an autopilot `work`/`close` worker or an escalation (unblock/deconflict/resolve/repair) session — as opposed to any external session a human or non-launcher path started. Its positive provenance makes it count in `plan`/`inflight` drain evaluation. Avoid: managed session, keeper-owned session, tracked session (that's broader).
## Crash restore
- **Generation**: One tmux server boot — every window and agent observed between a server start and its death, the cohort crash-restore scopes to. One boot carries exactly one keeper-stamped identity, so a probe-format change can never split it in two. Avoid: server epoch, boot id, killed cohort (that is the fallback derivation model, not the concept).
- **Restore**: Relaunching a dead generation's agent tabs so each re-attaches to its exact prior conversation, proven per tab from attach evidence — window creation alone never counts as restored. Avoid: revive (that is the runnable dump-script artifact), resurrect, respawn.
- **Harness resume**: One Supported harness re-attaching to its own persisted session — the per-tab primitive a restore drives — by its resume target: the Claude session id or Pi session id, where empty means not-resumable and a display title is never a resume key. Distinct from the Resume cursor (a fold checkpoint). Avoid: reconnect, reload, session name (for the key).
- **Refuse-live**: The resume-time gate that never re-attaches a currently live session — liveness is pid + start-time identity. Interactive resume refuses; response-bearing `agent run --resume` pins the exact live Partner, sends once over its existing Agent Bus inbox, and captures after the injected-message transcript boundary without another Harness writer. Avoid: force-resume, live takeover, double-attach.
## Frame stream and supervision
- **Frame**: One rendered snapshot of a viewer's body text, minted only when the rendered content actually changes; the unit the frame stream emits and a sidecar triple records. Avoid: screen, repaint, delta (that is the coarse tail's unit).
- **Stale-frame state / Paint watchdog**: The Stale-frame state is a live viewer's freshness presentation axis — composing with the connection axis, never mirrored from it — that renders a body-region red banner stamping a held frame's age past a short visible-switch debounce, freezes liveness affordances, and clears only on a proven fresh frame (an accepted daemon frame advancing the viewer's accepted-frame observable and forcing a full paint even when byte-identical, unlike a socket-open or local repaint). The Paint watchdog is the purely in-viewer, divergence-gated detector that trips only when the out-of-band daemon fold cursor advances while zero frames are accepted across its window — the connected-but-not-painting wedge no socket heartbeat sees — then renders the state and self-heals by resubscribing (a viewer-socket reconnect: the viewer re-dialing its own daemon UDS reconnect-forever, DISTINCT from Harness resume), never tripping an idle pane and emitting no synthetic event or process exit. Avoid: disconnected pill, spinner, heartbeat, liveness probe, Harness resume.
- **Resume cursor / Coverage verdict**: The Resume cursor is the fold-cursor checkpoint stamped on frames and trailers so a later chunk anchors where the last left off — non-unique, never a per-frame id. The Coverage verdict is the trailer's honest completeness claim — continuous only when one uninterrupted run provably dropped nothing, gap_possible otherwise. Avoid: offset, timestamp cursor, seq, gapless guarantee, completeness flag.
- **Hyper mode**: The supervision mode that consumes one bounded frame chunk per pass and judges each change as a human proxy — truthful, legible, stable — filing UI defects rather than editing renderers. Avoid: frame mode, firehose mode, vigilant mode.
## Install, reload, and daemon liveness
- **Shared-source leaf**: A Supported harness's global-instruction discovery path (for example `~/.claude/CLAUDE.md` or `~/.pi/agent/AGENTS.md`), re-asserted as a symlink into keeper's one `system/shared/AGENTS.md` source on every launch; healed by deleting the source, never the leaf, and distinct from the repo-root `AGENTS.md -> CLAUDE.md` convention symlink. Avoid: config leaf, stow target, dotfile link.
- **Load surface**: The set of checked-in files the resident daemon process actually loads — what the reload fingerprint hashes and the boundary test encloses via the checked-in **roots manifest** declaring its roots through one seam, so the hashed and enforced boundaries cannot disagree; per-invocation code (CLI, hooks, skills) sits outside it. Avoid: footprint, source tree, codebase, allowlist, path list, fingerprint config.
- **Daemon boot / Native-crash attribution**: A Daemon boot is one keeperd process lifetime, recorded in the restart ledger under a `boot_id`; Native-crash attribution is bounded platform crash-report evidence matched to that boot by pid, launch identity, and lifetime, never inferred or folded. Avoid: generation (a tmux server boot), instance, run, attribution (unqualified), crash guess.
- **Single-instance lock**: The kernel flock on the dedicated `keeperd.lock` file main acquires before opening the DB, making a second concurrent daemon impossible rather than merely detectable; the kernel releases it on process death, so a stale lock cannot exist. Avoid: pid file, sock lock (the server worker's separate per-socket lock file), mutex.
## Panels and launch triples
- **Partner**: A pair session that terminates with an envelope deliverable, is resumable by name, and while live admits one exact-identity response-bearing Bus artifact at a time; acknowledgement is not an answer, and capture begins only after the matching injected-message transcript boundary. Avoid: panel leg, worker, live target.
- **Handoff**: A delegated session carrying an event-sourced Brief whose caller owns the mandate; it parks under the receiving workflow by default, while explicit capture requests an autonomous terminal deliverable. Avoid: task ticket, partner, answer envelope.
- **Launch triple**: The context-free `<harness>::<model>::<effort>` token naming one launchable Claude or Pi configuration — the harness-native model id carried verbatim, the effort translated to the harness's own axis; every well-formed triple is launchable, enumerated by the matrix but never gated by it. A capability excluded from the worker cohort still enumerates as a launch triple — launch-only is per-capability, not per-provider. Avoid: preset, virtual preset, profile, model alias.
- **Panel**: A named, ordered selection of launch triples convened to answer one question in parallel, each member blind to the others, with a judge fusing the answers; a duplicated member is a distinct leg. Avoid: ensemble, quorum, committee.
- **Panel request**: One atomically admitted invocation carrying an opaque identity and an immutable argument digest; retries address that identity, while a later intentional invocation receives a new one even when its question is identical. Avoid: slug, prompt hash, Task call, attempt.
- **Panel run**: The full owned lifetime of one Panel request — member fan-out, judgment, cancellation, and cleanup — represented by one durable run directory and one monotonic result/cancellation outcome; cleanup progress is an orthogonal status that must settle before the outcome publishes successfully. Avoid: panel, batch, session, generation.
- **Panel cleanup status**: The durable `pending`/`failed`/`settled` state of exact resource teardown after a Panel run records cancellation intent; `failed` remains automatically reconcilable, and only `settled` proves every owned resource absent. Avoid: terminal outcome, cancellation result, wrapper death.
- **Run-control artifact**: The durable per-launch record binding one opaque run identity to its exact socket-qualified Tmux teardown command and cleanup state, consumed by cancellation and reconciliation without rediscovery from titles, PIDs, or window indexes. Avoid: pidfile, kill hint, window locator.
- **Panel leg**: One roster member's owned execution within a Panel run, including its bounded launch attempts and terminal result. Avoid: panelist, provider leg, member process, pane.
- **Task invocation**: One foreground call to the harness-neutral `Task(subagent_type, description, prompt)` tool contract, selecting a typed agent and returning its final result; distinct from a Plan Task on the Board. Avoid: task, agent run, worker dispatch.
