## Doc & comment style

The canonical comment + doc discipline (default to no comments, WHY-only, never provenance/narration, prune on touch, protected-comment allowlist, docs prune never append-only) lives once in `template/agents/worker.md.tmpl`'s `## Doc & comment discipline` block — every prompt surface echoes it; never fork a divergent wording. **Never write backward-facing advice.** Docs, comments, skills, agent prompts, and diagram labels describe the system as it is *now* — never by reference to what was removed, renamed, or previously true. No "X used to…", "formerly Y", "replaces the prior Z", "no longer carries…", "(deleted in fn-N)", "(now removed)", or a feature defined only by negating a dead one. State the present-tense fact; if a constraint matters, phrase it as a forward rule ("there is no `--no-push` flag"), not as a tombstone. The two sanctioned exceptions are `docs/adr/` — the typed home for resolved decision history, alongside commit messages — and the "Removed verbs (do not re-add)" guardrail list below, which exists precisely to carry history.

## Convention Divergences

These diverge from standard arthack conventions:

- **`cat` is format-free** — always emits raw markdown to stdout regardless of `--format`. FormattedGroup auto-injects `--format` so the flag appears in `cat --help`, but the cat verb ignores it.
- **`validate` envelope is non-`success`** — emits `{"valid": bool, "errors": [...], "warnings": [...]}` instead of the standard `{"success": bool, ...}`, exiting 1 on `valid: false`. Routes through the formatter directly (not `emit()`) to preserve the shape. On a fresh `--epic` stamp it merges the `plan_invocation` provenance INTO this one envelope — still exactly one top-level JSON value, never a second document.
- **Bare verb subcommand names** — `init`, `status`, `claim`, etc. instead of arthack's `verb-noun`. Established in the spec and referenced by orchestration scripts; do not rename.
- **Single implementation** — the compiled TypeScript dispatcher under `src/` is the sole runtime, reached in-process via `keeper plan`. The bun:test suite under `test/` is the living conformance surface.

## Commit behavior

Every mutating verb auto-commits its own data-dir scope inline at `output.emit()` (`chore(plan): <op> <target>`); a success envelope on stdout means the commit landed, and callers never wildcard-commit plan state.

## Data directory

The convention data dir is `.keeper/`. **Resolution and write-back live in one seam — `src/state_path.ts`** (`DATA_DIR = ".keeper"`, the single member of `DATA_DIR_NAMES`). Reads, detection, and write-back all target `.keeper/`; a root with no `.keeper/` yet defaults to minting one on the first write.

## Validation marker

`last_validated_at` is a strict **arm-exclusive one-way latch** — autopilot's dispatch gate (readiness predicate 2 blocks a null-marker **ghost**, rendered dashed, and ranks above the dep checks). `normalizeEpic` defaults it `null` and `scaffold` mints a fresh epic still null, so a freshly-scaffolded epic is a ghost by construction. Exactly three writer classes touch the field:

- **Arm (null → timestamp):** ONLY `validate --epic` / `armEpicValidated` — the trailing arm the create (`/plan:plan`), defer (`/plan:defer`), and close (`/plan:close` close-finalize) flows run after all deps are wired. close-finalize arms in-process via the non-exiting `armEpicValidated` seam (never the CLI `validate` verb's `process.exit` wrapper, which would corrupt its terminal envelope). Idempotent — an already-armed epic is a pure no-op.
- **Un-arm (timestamp → null):** ONLY `epic invalidate` and `refine-context --invalidate` (the `--invalidate` flag flips the otherwise read-only verb into a conditionally-mutating one, following `validate --epic`'s precedent).
- **Everything else:** the 12 verbs in `src/integrity_gate.ts::INTEGRITY_GATE_VERBS` (canonical list — do not duplicate here) run their post-write integrity gate but NEVER read or write the marker, so a ghost stays a ghost and an armed marker stays byte-identical through any interleaving of scaffold / cell-select / dep-wiring. Their `updated_at` bump is unchanged. Non-members include `done`, `claim`, `block`, `unblock`, `epic close`, `set-branch`, `set-title`.

`validate --epic` and `refine-context --invalidate` are both conditionally-mutating verbs: when the flag is absent or the precondition is already satisfied the verb is read-only and emits no commit; when the flag fires and a write is needed, the runner manually invokes `auto_commit_from_invocation` to land a single `.keeper/` commit. `validate --epic` writes the stamp BEFORE its auto-commit — on commit failure the stamp persists on disk and a re-run short-circuits silently; the next mutating verb's auto-commit sweeps the dirty file. Either path emits exactly one top-level JSON value on stdout (the `plan_invocation` merges into the custom envelope, never a second document).

**Read-only / inspection verbs emit exactly one top-level JSON value** and no `plan_invocation` trailer — a second root breaks `json.load` ("Extra data") and `jq`. A conformance guard (`test/verbs-readonly.test.ts`) enforces it by parsing roots off stdout, not counting lines.

## Skills and agents

`plan:*` skills live under `skills/`; the auditor/planner/scout/panel-runner/panel-judge agents live in `agents/`, while the workers render one self-contained `work` plugin per `{model × effort}` cell (sourced from `subagents.yaml`) under `workers/<model>-<effort>/` — manifest name `work`, agent name `worker`. The launcher selects the matching cell at dispatch via `--plugin-dir` (resolved producer-side from the task's `{model, effort}`), so `/plan:work` spawns the constant `work:worker` and the SubagentStop matcher is `^work:worker$` (fires cross-plugin). The orchestrator is content-blind, and `/plan:close` runs the audit inline before the irreversible `epic close` — one of five `CloseOutcome` members (`closed_clean` / `closed_with_followup` / `fatal_halt` / `partial_followup` / `followup_blocks_close`). `followup_blocks_close` is the **blocking follow-up gate**: when the audit's follow-up corrects a consumer-observable flaw the source establishes and exposes (verdict `blocks_closing: true`), `close-finalize` mints and arms the follow-up but leaves the SOURCE epic OPEN, holding it and every dependent until the follow-up lands, then a re-dispatched closer adopts it — a source held open under this gate is BY DESIGN, never a stuck board to force-close. The gate's follow-up substitutes the source's still-resolving deps for the source itself (a follow-up→source edge deadlocks), all pointing at already-done epics, so **`scaffold`'s epic-dep validator is deliberately status-blind** — it checks a dep resolves to exactly one project, never its status; that carve-out is load-bearing and tested, never add a status check to it. A policy-flagged task also runs a **per-task audit gate** on the same block machinery — no verb: its worker parks `blocked` with an `AUDIT_READY` reason after committing instead of stamping done, `/plan:work` audits that one commit set content-blind and unblock-resumes it (or rewrites the reason to `AUDIT_SEVERE` for the existing escalation), and the finding artifact under `state/audits/<epic_id>/tasks/<task_id>.json` is what the close audit fingerprint-links — never a re-added `audit-guard` / `epic auditor-done`. The plugin's `hooks/` layer enforces the content-blind orchestrator contract mechanically: a PreToolUse commit hard-deny, a SubagentStop worker guard, and a Stop checklist guard keep all implementation work inside the worker subagent. Separately, the keeper plugin's `PreToolUse(Bash)` branch-guard hook hard-denies a worker subagent from git branch create/switch and every mutating `git stash` (list/show/create allowed), enforcing the worker "work in place" invariant mechanically. The sanctioned answer to "is this failing test pre-existing at my base commit?" is `keeper baseline <base sha> --wait`, never a stash-and-rerun.

`/plan:hack` (`skills/hack/SKILL.md`) is a hand-authored STATIC skill — no `.tmpl`, no `.managed-file-dont-edit` sidecar — that investigates a request, answers in the right shape, and routes or executes the next move. It is slash-only (`disable-model-invocation: true`). Once an epic lands the wrap-up is quiet by default: autopilot dispatches and completes all plan work, so the planning flow never proactively surprise-launches execution mid-plan — no `/plan:work`, no unsolicited surfacing of the operator skills. The `keeper:dispatch` / `keeper:autopilot` operator skills are model-invocable and may be reached on clear user intent, but never from the planning flow on its own. The skill arms a `keeper:await` only on a positive wait-then-act call (and asks once when ambiguous, else stays quiet), and runs an always-on close-signal that speaks a single closing line only when nothing in the thread is left. It bakes six shared `keeper prompt` snippets byte-verbatim (`engineering/keeper-history-forensics`, `engineering/domain-docs`, `source-dirs/docs-dir-and-gist-open`, `engineering/escalate-inline-or-plan`, `engineering/commit-via-keeper-default`, `engineering/landed-vs-complete`), each wrapped in a `<!-- BAKE:BEGIN keeper prompt render <ref> -->` / `<!-- BAKE:END … -->` guard whose region must equal `keeper prompt render <ref>` byte-for-byte; the drift gate (`bun scripts/vendor-corpus.ts --check`, also run in the prompt test suite) re-renders each guard against the vendored corpus and fails on any drift. The keeper operator skills carry the same snippet scheme's sibling — a `<!-- POINTER: keeper prompt render engineering/orient -->` marker above a skill-specific summary that is deliberately NOT verbatim and the byte-equality gate skips. Renders resolve against the corpus subset vendored under `plugins/prompt/corpus/` (provenance in its `vendor.lock`).

`/plan:unblock`, `/plan:deconflict`, and `/plan:repair` (`skills/{unblock,deconflict,repair}/SKILL.md`) are the escalation-session skills autopilot boots as `unblock::<task>` / `deconflict::<epic>` / `repair::<repo_token>`: unblock stays diagnosis-only (board + repo inspection, never Edit/Write), while deconflict (one epic's merge fan-in) and repair (a repo-scoped shared-base breakage spanning every epic on that repo) are write-capable and land their own commits. All three are role-keyed by the launch-injected `KEEPER_ESCALATION_ROLE` env marker, which the keeper plugin's `PreToolUse(Bash)` escalation-guard hook enforces mechanically as a per-role command-family allowlist — it is a keeper-hook-read marker, never a `KEEPER_PLAN_*` variable, so it does not appear in the Environment variables table below.

`/plan:prompt` (`skills/prompt/SKILL.md`) is a hand-authored STATIC skill — no `.tmpl`, no `.managed-file-dont-edit` sidecar — that polishes a raw prompt toward a named maturity rung (headline/blurb/note/memo/brief/spec) through a batched loop: each turn recomputes a filled/total slot meter and makes one clustered move (an approvable change-set, an `AskUserQuestion` batch, or an explore), and at first ready offers a Ship it / Keep polishing / Grow a rung fork. Slash-only (`disable-model-invocation: true`), read-only wrt the repo, clipboard-only export — it writes nothing to disk.

`/plan:cell-review` (`skills/cell-review/SKILL.md`) is a hand-authored STATIC skill — no `.tmpl`, no `.managed-file-dont-edit` sidecar — that grades worker-cell selection out-of-band, long after close, and is the sole grading surface (no close-time beat, no board flag). Its work-list is the set difference of committed `selection-audit-briefs/` minus committed `selection-reviews/` — the authoritative watermark, never a `~/docs` file or wall-clock. It fans the blinded `plan:selection-auditor` over each epic's committed brief (enriched with read-only keeper.db difficulty proxies, blind to the selector's confidence/rationale), lands each verdict through `selection-review-submit` stamped with `rubric_version`/`judge_model_version`/`prompt_hash` (a re-grade on a version change passes `--force`; commit-then-advance means a crash resumes from the set difference), then computes `config_hash` cohort rates with Wilson intervals and refuses any proposal below a minimum cohort count. It writes human-facing reports under `~/docs/selection-reviews/` (markdown bodies only — sidecars belong to the hooks) and PROPOSES exact `model-selector.yaml` guidance edits it never writes: `hand_tuned` and the drift-gated `efforts:`/`models:` blocks are authored only by a human or `/plan:model-guidance`. Slash-only (`disable-model-invocation: true`).

## Removed verbs (do not re-add)

`start` (→ `claim`), `next`, `activity`, `scout`, `interview`, `config show`, `draft from-close`, `epic publish`, `epic create --draft`, `epic.draft` field, `epic auditor-done`, `migrate-audit-grandfathering`, `audit-guard`, `close-context`, `epic close --audit-required`/`--no-audit-required`, `epic.auditor_done_at` field, `task create`, `task set-spec`, `task set-deps`, `epic set-plan`, `dep add` (whole `dep` group), `gravity`, `classifier` (agent + `skills/close/classifier/` schema — the close-planner absorbs vet/cull/merge + verdict authoring), `epic followup-of` (the partial-follow-up completeness check is ported into `close-finalize`), `epic set-snippets`, `epic set-bundles`, `task set-snippets`, `task set-bundles`, `scaffold --snippets`/`--bundles` flags, `epic queue-jump` + the `/plan:next` skill + `epic.queue_jump` field (board priority lives on the autopilot surface, never plan metadata), `task set-tier` (the `tier` + `model` axes are chosen at plan/refine/select time and carried on the task JSON — a scaffold stamps the mechanical default cell, then the post-scaffold selector batch-overwrites it via `assign-cells`; there is no incremental single-task set-tier / set-model verb, only that whole-epic batch write), `selection-review` (the board-visible selection-review flag; grading a closed epic's cells is a human-invoked, out-of-band skill reading the committed `selection-audit-briefs/` + `selection-reviews/` datasets, never a close-time board flag). Their structural effects ride `scaffold` (mint a fresh epic + tasks), `refine-apply` (add/rewrite on an existing epic), and `assign-cells` (batch-overwrite a ghost epic's `{tier, model}` cells in the post-scaffold select window, landing a git-committed selection sidecar in the same auto-commit). Epic-level deps keep `epic add-dep` / `epic add-deps` / `epic rm-dep`.

The no-incremental-mutation stance above is NOT a no-delete stance. `keeper plan epic rm <epic_id>` is the sanctioned — and only — delete verb: it unlinks every artifact an epic owns (epic JSON, every child task JSON, epic + task spec markdowns, runtime state, lock files) and auto-commits the deletions into the owning project's `.keeper/` via the standard envelope path. Resolves cwd-then-global with a `--project` escape, guards `in_progress` / locked tasks behind `--force`, and supports `--dry-run`. Companion guard on the create side: `scaffold` rejects a same-slug sibling epic up front with `duplicate_epic` (naming the existing id + status); `--allow-duplicate` is the explicit escape hatch.

## Environment variables

- `KEEPER_PLAN_ACTOR` overrides actor identity.
- `KEEPER_PLAN_NOW` overrides the clock source for all timestamp stamping in `%Y-%m-%dT%H:%M:%S.%fZ` format; any conforming implementation must honor it.
- `KEEPER_PLAN_WORKTREE` overrides the worktree-mode lane PATH for a worker's `target_repo` ONLY (via `expectedWorkerCwd`): when set non-empty it wins over the ordinary fallback chain so concurrent lanes don't collide in the shared main checkout. Plan STATE always resolves to the primary repo from `epic.primary_repo`, never the lane — every runtime-overlay / close-audit verb routes its state-bearing context through the ONE central seam `resolvePlanStateContext` (`src/project.ts`; close-preflight / close-finalize thread the equivalent inline `contextForRoot(primaryRepo)` precedent), keying physical ownership on the committed `epic.primary_repo` FIELD, not cwd or where defs sit. `KEEPER_PLAN_WORKTREE` moves only `targetRepo`, never state. Producer-only (set at the worker's child boundary by autopilot worktree mode, realpath-normalized); the PATH is NEVER written to the event log as a fold key (it embeds a provision-time dirhash and is torn down at finalize). Empty/unset falls through to the fallback chain unchanged.
- `KEEPER_PLAN_WORKTREE_BRANCH` is the sibling marker env carrying the lane BRANCH (`keeper/epic/<id>[--<task>]`). Unlike the path above it IS a captured durable value: the hook reads it at SessionStart into `events.worktree`, folded set-once onto `jobs.worktree` (the branch survives `git worktree remove`/`move` where the path dangles). Producer-only, always-emitted (empty on serial launches so a reused tmux session never inherits a stale branch).
